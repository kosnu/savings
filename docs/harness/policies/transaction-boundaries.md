# Transaction Boundaries

1つのユーザー操作が複数のデータ作成・更新・削除を含む場合は、片方だけ成功した状態がユーザー体験やデータ整合性を壊さないかを確認します。

片成功を避ける必要がある場合は、Design / Planで同一操作単位を明示します。実装手段は既存構成と制約に合わせて選びます。

## 判断基準

次のいずれかに当てはまる場合は、同一操作単位として扱うことを検討します。

- 片方だけ成功すると、ユーザーに成功状態を誤認させる
- 片方だけ成功すると、後続画面や集計結果が要件と矛盾する
- ユーザーが1つの保存・作成・削除操作として認識する
- 失敗時に個別復旧よりも全体失敗として扱う方が自然
- 片成功の検知や補償処理が複雑になる

## Design / Planで決めること

- どのデータ変更を同一操作単位に含めるか
- 期間や履歴を持つデータでは、対象月や対象日を決める入力を同一操作単位に含めるか
- どの層で境界を作るか
- 操作の実行境界と、入力値へ適用するルールの所有境界をどの層が担うか
- 片成功が起きた場合のユーザー影響
- 採用しない案と理由
- 残る競合リスクやスコープ外の補償

## 実行境界とルール所有境界

操作の原子性を保証する実行境界と、validation、正規化、上限、既定値などのルール所有境界は分けて判断します。

RPCやdatabase functionを実行境界に選んでも、その操作に関わるすべてのルールをDBが所有することにはなりません。ルールを配置する層は、その実行手段ではなく、ルールを守る理由と適用範囲で決めます。

- 呼び出し元に関係なく守る必要があり、データ整合性、参照整合性、認証、権限、または原子性を保護するルールは、APIまたはDBの境界で保証します。
- 特定のアプリケーションやユーザー体験に固有で、保存層がより広い値を正しく保持できる入力制約、正規化、上限、既定値は、そのアプリケーション層が所有します。
- アプリケーション層が所有する値ルールは、適用後の値をRPCやAPIへ明示的な入力として渡します。RPCやDB側で同じルールを再定義しません。
- 実行境界を選ぶ都合だけでルール所有境界を移す必要がある場合は、Design / Planで理由と影響範囲を明示します。根拠をRequirementsまたは正本ルールへ追跡できない場合はStopします。

## 実装境界の候補

- DB transaction / database function
- RPC / API endpoint
- アプリケーション層の明示的な更新順序と失敗処理
- 補償処理または再試行

DB constraint、trigger、RPC、API追加などが既存の制約やスコープを超える場合は、Build / Verifyで勝手に追加せずStop条件として扱います。

時系列状態を保存する操作では、DB側の現在日時で対象期間を補完して、read側とwrite側の期間基準を分離しないようにします。
