# Learning Extraction

学びとは、レビューコメント、検証結果、運用上の指摘、変更された制約などから抽出でき、タスクをまたいで再利用すべき知見です。

個別タスク内で完結する不具合、テスト失敗、型、lint、呼び出し側調整は、それだけでは学びにしません。タスクコンテキスト、横断ルール、ポリシー、監督制約、恒久的な検証期待を変える場合だけ学びとして扱います。

## 整理先

- タスクコンテキスト: 現在のタスク固有の意図、スコープ、制約、成功条件、監督観点、または新しい体験や価値を定義するRequirements材料
- 暫定監督ルール: 限定されたプロダクト段階のあいだ、複数の関連タスクへ適用する期限付きの判断基準
- ルール・ポリシー: 期限を設けず、複数タスクで継続して適用する判断基準や禁止事項
- 除外: 既存のタスクコンテキストまたは正本ルールだけで判断が完結しており変更が不要な知見、根拠不足、または再利用しない知見

タスクコンテキストとルール・ポリシーの両方が必要な場合は、タスクコンテキストで体験、価値、スコープを定義し、継続的な判断原則は正本ルールを参照します。同じ内容を両方へ重複して記載しません。

## 分類原則

- 現在のタスクにだけ必要な情報はタスクコンテキストへ置きます。
- 新しい体験や価値はタスクコンテキストへ置きます。その体験を継続的に維持する判断原則が必要な場合は、正本ルールも作成または更新し、タスクコンテキストから参照します。
- 複数の関連タスクへ適用するが、特定のプロダクト段階で失効する制約は、恒久ルールへ混ぜず暫定監督ルールとして扱います。
- 暫定監督ルールには、適用対象、判断基準、失効条件、失効後に定義すべき体験または正本ルールを明記します。
- ルール・ポリシーには、期限付きの事情や現在の未対応を固定化しません。正常パターンと責務を直接定義し、場当たり的な回避策や迂回的な手順をルール化しません。
- 学びの根拠に将来の構想が含まれていても、対応する新規タスクや体験が定義されていない場合は、先回りして別タスクのタスクコンテキストを作りません。

## 実行文脈

- AI Driven Development: Ship完了後にユーザーがLearn skillを手動実行し、成果物フィードバックをRequirementsの材料となるタスクコンテキストまたはルール・ポリシーへ整理する。
- harness-task: レビューや検証で得た学びを、harness-taskのInputとなるタスクコンテキスト、暫定監督ルール、もしくはルール・ポリシーへ整理する。明示されたスコープ内のルール更新はタスク内で実施できる。

Learn skillは学びの抽出と整理に特化した導線ですが、学び抽出を独占しません。`harness-task`も作業中に学びを抽出し、スコープ内のルール更新またはタスクコンテキストのhandoffを行えます。

通常タスクのレビューコメントでは、現在のタスクで必要な修正を先に判断します。学び抽出は必要な修正の代わりではなく、Build / Verify完了後であることだけを理由に修正をStopさせません。

## 根拠の境界

前回実装コード、前回UI挙動、現在のdiff形状、前回実装固有の設計判断を、タスクコンテキストやルールの正本にしません。学びは、元のIssueまたはタスクコンテキスト、明示的なフィードバック、検証・運用結果、既存の正本ルール、監督制約から説明します。

## Stop条件

- 学びの根拠が曖昧
- 反映先のタスクまたは正本ルールが特定できない
- 暫定監督ルールの適用対象または失効条件が特定できない
- 学びを説明するために前回実装固有の事実を正本化する必要がある
- 選択したrule-mapサブグラフと矛盾する
