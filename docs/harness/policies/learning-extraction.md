# Learning Extraction

学びとは、レビューコメント、検証結果、運用上の指摘、変更された制約などから抽出でき、タスクをまたいで再利用すべき知見です。

個別タスク内で完結する不具合、テスト失敗、型、lint、呼び出し側調整は、それだけでは学びにしません。タスクコンテキスト、横断ルール、ポリシー、監督制約、恒久的な検証期待を変える場合だけ学びとして扱います。

## 整理先

- タスクコンテキスト: Requirementsを作る材料、またはharness-taskへ渡すInputとなる意図、スコープ、制約、成功条件、監督観点
- ルール・ポリシー: 複数タスクで継続して適用する判断基準や禁止事項
- 除外: タスク内で修正済み、既存ルールに表現済み、根拠不足、または再利用しない知見

タスクコンテキストとルール・ポリシーの両方が必要な場合は、タスクコンテキストから正本ルールを参照し、同じ内容を二重管理しません。

## 実行文脈

- AI Driven Development: Ship完了後にユーザーがLearn skillを手動実行し、成果物フィードバックをRequirementsの材料となるタスクコンテキストまたはルール・ポリシーへ整理する。
- harness-task: レビューや検証で得た学びを、harness-taskのInputとなるタスクコンテキスト、もしくはルール・ポリシーへ整理する。明示されたスコープ内のルール更新はタスク内で実施できる。

Learn skillは学びの抽出と整理に特化した導線ですが、学び抽出を独占しません。`harness-task`も作業中に学びを抽出し、スコープ内のルール更新またはタスクコンテキストのhandoffを行えます。

通常タスクのレビューコメントでは、現在のタスクで必要な修正を先に判断します。学び抽出は必要な修正の代わりではなく、Build / Verify完了後であることだけを理由に修正をStopさせません。

## 根拠の境界

前回実装コード、前回UI挙動、現在のdiff形状、前回実装固有の設計判断を、タスクコンテキストやルールの正本にしません。学びは、元のIssueまたはタスクコンテキスト、明示的なフィードバック、検証・運用結果、既存の正本ルール、監督制約から説明します。

## Stop条件

- 学びの根拠が曖昧
- 反映先のタスクまたは正本ルールが特定できない
- 学びを説明するために前回実装固有の事実を正本化する必要がある
- 選択したrule-mapサブグラフと矛盾する
