# Requirements / PRD: 月単位画面で前月・翌月へ移動しやすくする

## 背景と課題

月単位で支払い、予算、サマリーなどを確認する画面では、対象月を変更して月ごとの状態を見比べる操作が発生する。Issue #1492 は、月変更をやりやすくするために、前月・翌月へ送る操作を含む月送り機能の Requirements / PRD 作成を求めている。

現行の `/payments` では、支払い一覧、カテゴリ絞り込み、月次サマリーが同じ対象月の文脈で表示される。対象月が URL search params の `year` / `month` にない場合は現在月が補完され、サマリー上の月選択導線から任意の年月を選べる。一方で、連続して前月・翌月を確認したい場合、毎回年月を選び直す必要があり、月をまたいだ支払い状況や予算状況の比較に手間がかかる可能性がある。

この要求は Issue #1492 を入力として整理する。Issue は最終仕様ではなく、Requirements / PRD のための意図、境界、成功条件として扱う。

## Current State / Current Gaps

- `/payments` は支払い一覧、カテゴリ絞り込み、月次サマリーを表示する。
- `/payments` の対象月は `year` / `month` search params で表される。
- `year` / `month` が不足している場合、認証済みユーザーでは現在年月が補完される。
- 月次サマリーには対象月を選ぶ導線がある。
- 月次支出合計は対象月の1日以上、翌月1日未満の支払いを合計する。
- 月次予算の対象月は年と月の組み合わせで扱われる。
- 支払い一覧にはカテゴリ絞り込み条件があり、月変更時に矛盾させてはいけない。
- 現行導線だけでは、前月・翌月へ続けて移動したいときに、月を選び直す操作が必要になる。

## Future Behavior

ユーザーは月単位の画面で、現在の対象月を理解したまま、前月・翌月へ簡単に移動できる。月を変更すると、支払い一覧、予算使用状況、カテゴリ別サマリーなど対象月に依存する情報が新しい対象月の文脈へ切り替わったことを理解できる。

任意の年月を選ぶ既存導線は維持する。前月・翌月への移動は、既存の月選択導線を置き換えるものではなく、隣接月へ短く移動したい利用シーンを補うものとして扱う。

## 対象ユーザーと利用シーン

- 対象ユーザー: savings の Web UI で月ごとの支払い、予算、サマリーを確認する認証済みユーザー。
- 月ごとの比較: 前月と今月、今月と翌月など、隣接月の支払い状況や予算状況を見比べたい。
- 連続確認: 対象月を1か月ずつ移動しながら、月次サマリーや支払い一覧の変化を確認したい。
- 絞り込み併用: カテゴリ絞り込みなど既存条件を使ったまま、対象月だけを前後へ動かしたい。
- 対象月確認: 操作後に、いま見ている支払い、予算、サマリーが何年何月の情報かを理解したい。

## ユーザーストーリー

- 認証済みユーザーとして、月単位画面で前月へすぐ移動したい。そうすることで、直前の月の支払い状況や予算状況を確認できる。
- 認証済みユーザーとして、月単位画面で翌月へすぐ移動したい。そうすることで、隣接する月へ戻る、または次の月の状態を確認できる。
- 認証済みユーザーとして、対象月を変更した後も、いま何年何月を見ているかを理解したい。
- 認証済みユーザーとして、カテゴリ絞り込みなどの既存条件を使っているときも、月送り操作で対象月だけを変更したい。
- 認証済みユーザーとして、任意の年月を選びたい場合は既存の月選択導線を引き続き使いたい。

## スコープ

### 対象

- `/payments` の月単位表示における対象月変更体験。
- 対象月表示、既存の年月選択導線、前月・翌月へ移動する月送り操作の関係整理。
- 月変更後に、支払い一覧、月次支出合計、月次予算使用状況、カテゴリ別サマリーなど対象月に依存する情報が切り替わる期待挙動。
- カテゴリ絞り込みなど、既存の月以外の検索条件と月送り操作の関係。
- 後続 Design / Plan で決めるべき UI 形状、操作名、アクセシブルな名前、検証観点の整理。

### 対象外

- 月次予算ドメインの仕様変更。
- 支払い、カテゴリ、予算のデータ構造変更。
- 新しい集計ロジックの追加。
- 月以外の期間単位への拡張。
- DB、API、認証、権限、RLS、RPC の変更。
- Requirements / PRD 内での具体的なボタン配置、component、アイコン、色、variant、最終文言の固定。
- この Goal 内での Design Doc 作成、実装、commit、PR 作成。

## 機能要件

### FR-1: 対象月を識別できる

- ユーザーは月単位画面で、現在の対象月が何年何月かを理解できる。
- 対象月は、支払い一覧、月次支出合計、月次予算使用状況、カテゴリ別サマリーの文脈と矛盾しない。
- 対象月が URL search params など既存状態から決まる場合、その状態と画面上の表示がずれてはいけない。

### FR-2: 前月へ移動できる

- ユーザーは現在の対象月から前月へ移動できる。
- 1月から前月へ移動する場合は前年12月として扱う。
- 前月へ移動した結果、対象月に依存する支払い、予算、サマリーは前月の文脈へ切り替わる。
- 前月へ移動しても、月以外の既存検索条件は矛盾なく扱われる。

### FR-3: 翌月へ移動できる

- ユーザーは現在の対象月から翌月へ移動できる。
- 12月から翌月へ移動する場合は翌年1月として扱う。
- 翌月へ移動した結果、対象月に依存する支払い、予算、サマリーは翌月の文脈へ切り替わる。
- 翌月へ移動しても、月以外の既存検索条件は矛盾なく扱われる。

### FR-4: 既存の任意年月選択導線を維持する

- ユーザーは既存の月選択導線で任意の年月を選べる。
- 月送り操作は、任意年月選択導線を置き換えない。
- 任意年月選択と前月・翌月移動は、同じ対象月状態を更新する操作として矛盾しない。

### FR-5: 月以外の既存条件を壊さない

- カテゴリ絞り込みなど、月以外の検索条件を使っている場合でも、月送り操作は対象月だけを変更する。
- 月送り操作後も、月以外の検索条件が意図せず解除、変更、追加されてはいけない。
- 支払い詳細など `/payments` 配下の既存 URL 状態と矛盾する場合は、Design / Plan で扱いを決める。

### FR-6: 既存ドメイン仕様を維持する

- 月送り操作は支払い、カテゴリ、月次予算のデータ構造を変更しない。
- 月送り操作は月次支出合計、カテゴリ別支出合計、月次予算使用状況の計算ルールを変更しない。
- 月送り操作は過去月、現在月、未来月の保存可否や予算作成可否を変更しない。
- 月送り操作は追加の期間単位、期間範囲、比較軸を導入しない。

## 非機能要件と制約

- 既存 Web UI の文脈に合う形で改善する。
- Requirements では具体的なボタン配置、アイコン、component、色、variant、最終文言を固定しない。
- 対象月変更は、ユーザーが理解できる1回の結果として扱う。
- 対象月変更後に古い対象月の情報が完了済み状態として見えてはいけない。
- 新規依存の追加はこの要求の前提にしない。
- DB、API、認証、権限、RLS、RPC の変更を前提にしない。
- Issue #1492 と選択 refs から追跡できない成功条件を追加しない。

- AC-2: 前月へ移動する操作を実行すると、対象月が1か月前へ変わる。
- AC-3: 2026年1月から前月へ移動すると、対象月が2025年12月になる。
- AC-4: 翌月へ移動する操作を実行すると、対象月が1か月後へ変わる。
- AC-5: 2025年12月から翌月へ移動すると、対象月が2026年1月になる。
- AC-6: 月変更後、支払い一覧、月次支出合計、月次予算使用状況、カテゴリ別サマリーなど対象月に依存する表示が新しい対象月の文脈で扱われる。
- AC-7: 月変更後も、カテゴリ絞り込みなど月以外の既存検索条件が意図せず失われない。
- AC-8: 既存の任意年月選択導線は引き続き使える。
- AC-9: 月送り操作によって、支払い、カテゴリ、予算データ構造、月次集計ロジック、月次予算ドメイン仕様は変わらない。

## Q&Aログ

- Q: Issue #1492 は最終仕様として扱うか？
  - A: 扱わない。Issue は Requirements / PRD の入力として扱い、要求、制約、受け入れ条件へ展開する。
- Q: 対象画面はどこか？
  - A: 優先対象は `/payments` の月単位表示である。ここでは支払い一覧、月次サマリー、カテゴリ絞り込みが同じ対象月文脈を持つ。ほかの月単位 UI が見つかった場合は、Issue #1492 の意図に照らして Design / Plan で追加対象か対象外かを確認する。
- Q: 月送り操作は既存の月選択導線を置き換えるか？
  - A: 置き換えない。任意年月選択は維持し、前月・翌月への短い移動導線を補う。
- Q: 前月・翌月操作の見た目や配置をこの PRD で決めるか？
  - A: 決めない。操作できること、対象月が変わること、既存導線と矛盾しないことを要求にし、具体的な UI は Design / Plan で決める。
- Q: カテゴリ絞り込みは月送り時に解除するか？
  - A: 解除しないことを期待する。月送りは対象月を変える操作であり、月以外の既存検索条件を意図せず変えない。
- Q: 未来月への移動可否を制限するか？
  - A: Requirements では新しい制限を追加しない。既存の表示、検索、月次予算ルールと矛盾する場合は Design / Plan で確認する。
- Q: 月次予算や支払いの保存ルールを変更するか？
  - A: 変更しない。今回の対象は月単位表示の対象月変更体験であり、保存ルールやデータ構造は対象外である。

## 技術的考慮事項

- `/payments` は支払い一覧、カテゴリ絞り込み、月次サマリーを同じページで扱う。
- 現行の対象月は `year` / `month` search params で表される。
- 対象月 search params が不足している場合、認証済みユーザーでは現在年月が補完される。
- 現行の月選択導線は、選択した年月を `/payments` の search params に反映する。
- カテゴリ絞り込みは `category` search param として扱われる。
- 後続 Design / Plan では、月送り操作がどの route の search params を更新するか、カテゴリ条件などをどう保持するか、支払い詳細など `/payments` 配下での扱いを決める。
- 後続 Design / Plan では、操作名、アクセシブルな名前、前月/翌月の年またぎ、現在月補完との関係、既存の任意年月選択導線との配置関係を決める。
- 後続 Build / Verify で Web アプリケーションコードを変更する場合は、AGENTS.md の Web verification batch が必要になる。

## Verification

## Stop 条件

- 月送り機能を入れる対象画面が複数解釈でき、PRD 上で優先範囲と対象外を決められない。
- 月送り操作が既存の月選択、カテゴリ絞り込み、URL 状態、データ取得境界と矛盾する。
- Issue #1492、oversight input、選択 refs から追跡できない要求や成功条件を追加する必要がある。
- 月次予算ドメイン、支払い、カテゴリ、予算データ構造、新しい集計ロジック、月以外の期間単位への拡張が必要になる。
- 新規依存、DB変更、API仕様変更、認証/権限変更、破壊的 git 操作、実装変更が必要になる。
- Requirements / PRD が選択したルールまたはポリシーに違反している、または違反の可能性がある。
