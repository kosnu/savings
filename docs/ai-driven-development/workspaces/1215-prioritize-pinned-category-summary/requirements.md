# Requirements / PRD: カテゴリ別サマリーでピン留めカテゴリを優先表示できるようにする

## 背景

支払い一覧では、月ごとの支払いを確認しながらカテゴリ別の合計金額も見られる。カテゴリ別サマリーは支出状況の把握に役立つが、全カテゴリを常に表示すると支払い一覧の表示領域を大きく使い、日々の支払い確認を妨げる。

ユーザーが重要と考えるカテゴリは、ユーザーや時期によって異なる。単にカテゴリ ID 昇順で表示するだけでは、確認したいカテゴリが初期表示に含まれない可能性がある。

この要求は GitHub Issue #1215 を入力として整理する。Issue は最終仕様ではなく、Requirements / PRD のための意図、境界、成功条件として扱う。

## 解決したい課題

ユーザーは支払い一覧を見ながら、重要なカテゴリの支出状況を素早く確認したい。

しかし、全カテゴリのサマリーが常に表示されると、確認したい支払い一覧の領域が圧迫される。また、ユーザーごとに重要なカテゴリは異なるため、固定順だけでは確認したいカテゴリが初期表示に出ない可能性がある。

## 期待する状態

支払い一覧のカテゴリ別サマリーは、初期表示では表示件数を抑えつつ、ユーザーが重要と考えるカテゴリを優先して確認できる。

ユーザーはカテゴリ作成・編集時にカテゴリをピン留めできる。ピン留めされたカテゴリはカテゴリ別サマリーの初期表示で優先される。初期表示に出ないカテゴリも、追加表示操作によって画面上で確認できる。

ピン留めできるカテゴリ数は最大 3 件に制限する。これにより、ユーザーが「ピン留めしたのに初期表示されない」状態を避ける。

## 対象ユーザーと利用シーン

- 対象ユーザー: 個人の支払い・貯蓄管理をこのアプリで行う認証済みユーザー。
- 利用シーン: 支払い一覧で月ごとの支払いを確認しながら、特に重要なカテゴリの支出額を素早く見る。
- 関連する現行体験:
  - 支払い一覧に月選択、月合計、カテゴリ別サマリーが表示される。
  - カテゴリ別サマリーはアコーディオン内に表示される。
  - カテゴリ別サマリーはカテゴリ ID 昇順で表示される。
  - 未分類支払いは `Unknown` として別集計される。
  - 設定画面ではカテゴリの作成、名前変更、削除、ピン状態の表示ができる。

## ユーザーストーリー

- 認証済みユーザーとして、支払い一覧で重要なカテゴリの合計金額を最初に確認したい。そうすることで、支払い一覧を大きく押し下げずに重点カテゴリの支出状況を把握できる。
- 認証済みユーザーとして、カテゴリ作成・編集時にカテゴリをピン留めしたい。そうすることで、自分が重視するカテゴリをカテゴリ別サマリーの初期表示に出しやすくできる。
- 認証済みユーザーとして、初期表示に出ていないカテゴリ別サマリーも必要なときに確認したい。そうすることで、表示領域を抑えつつ全カテゴリの集計も確認できる。

## スコープ

### 対象

- 支払い一覧のカテゴリ別サマリーの初期表示を最大 3 件にすること。
- カテゴリ作成・編集フォームでカテゴリのピン留め状態を設定できること。
- ピン留め登録数を最大 3 件にすること。
- ピン留めされたカテゴリをカテゴリ別サマリーの初期表示で優先すること。
- ピン留めカテゴリだけで初期表示枠が埋まらない場合、未ピン留めカテゴリも含めて最大 3 件まで補うこと。
- ピン留めがない場合、カテゴリ ID 昇順を基準に初期表示すること。
- ピン留め同士、未ピン留め同士の並び順はカテゴリ ID 昇順を基準にすること。
- 4 件目以降のカテゴリ別サマリーを、取得済みデータから画面上で段階的に表示できること。
- ピン留め状態をカテゴリ本体ではなく、カテゴリ表示設定として扱うこと。
- カテゴリ作成・編集とピン留め状態変更の片方だけが成功する状態を避けること。

### 対象外

- サーバーからカテゴリ別サマリーを追加取得するページング。
- カテゴリ別サマリー以外の一覧表示改善。
- カテゴリの並び替えをユーザーが任意に変更する機能。
- カテゴリ表示設定を Book 単位へ移行すること自体。
- Requirements / PRD 作成前の実装、Design Doc 作成、PR 作成。

## 機能要件

### FR-1: カテゴリ別サマリーの初期表示を最大 3 件にする

- ユーザーが支払い一覧のカテゴリ別サマリーを開いた直後、表示されるカテゴリ別サマリーは最大 3 件である。
- カテゴリ別サマリーが 3 件以下の場合、存在するカテゴリ別サマリーをすべて表示する。
- 初期表示件数の制限は画面上の表示件数の制限として扱い、取得済みカテゴリ別サマリーを失わせない。

### FR-2: ピン留めカテゴリを優先表示する

- カテゴリ別サマリーの表示順は、ピン留めカテゴリを未ピン留めカテゴリより優先する。
- ピン留めカテゴリ同士はカテゴリ ID 昇順で表示する。
- 未ピン留めカテゴリ同士はカテゴリ ID 昇順で表示する。
- ピン留めカテゴリが 3 件未満の場合、未ピン留めカテゴリをカテゴリ ID 昇順で追加し、初期表示を最大 3 件まで補う。
- ピン留めカテゴリがない場合、カテゴリ ID 昇順で最大 3 件を初期表示する。

### FR-3: カテゴリ作成時にピン留め状態を設定できる

- ユーザーはカテゴリ作成フォームで、新しいカテゴリをピン留めするかどうかを指定できる。
- ピン留めありで作成されたカテゴリは、カテゴリ表示設定としてピン留め状態を保持する。
- ピン留めなしで作成されたカテゴリは、カテゴリ別サマリーのピン留め優先対象にならない。
- ピン留めありのカテゴリ作成では、カテゴリ作成とピン留め登録の片方だけが成功する状態を避ける。

### FR-4: カテゴリ編集時にピン留め状態を変更できる

- ユーザーはカテゴリ編集フォームで、既存カテゴリのピン留め状態を確認できる。
- ユーザーはカテゴリ編集フォームで、既存カテゴリをピン留めまたはピン留め解除できる。
- ピン留め状態の変更後、設定画面とカテゴリ別サマリーの表示は最新のピン留め状態に基づく。
- カテゴリ名更新とピン留め状態変更を同時に扱う場合、片方だけが成功する状態を避ける。

### FR-5: ピン留め登録数を最大 3 件にする

- ユーザーがピン留めできるカテゴリは最大 3 件である。
- ピン留め登録数がすでに 3 件ある場合、未ピン留めカテゴリを追加でピン留めする操作は成功扱いにしない。
- 4 件目のピン留め登録に失敗した場合、ユーザーはピン留め登録数の上限に達しているため失敗したことを認識できる。
- ピン留め解除後は、別のカテゴリをピン留めできる。

### FR-6: 4 件目以降のカテゴリ別サマリーを追加表示できる

- 初期表示に含まれないカテゴリ別サマリーがある場合、ユーザーは `Show more` 相当の操作で残りを表示できる。
- 追加表示は取得済みのカテゴリ別サマリーを画面上で表示する操作として扱う。
- 残りを表示した後、ユーザーが未表示カテゴリの存在を誤認しない状態にする。
- `Show more` 相当の操作で残りを表示した後に、月変更や集計結果の変更が起きた場合、カテゴリ別サマリーは初期表示件数へ戻る。

### FR-7: 失敗時にユーザーが失敗を認識できる

- カテゴリ作成時のピン留め状態保存に失敗した場合、ユーザーは画面上で失敗を認識できる。
- カテゴリ編集時のピン留め状態変更に失敗した場合、ユーザーは画面上で失敗を認識できる。
- ピン留め登録数の上限に達しているため失敗した場合、ユーザーは汎用的な作成・更新失敗ではなく、上限到達が理由であることを認識できる。
- 失敗したピン留め状態変更を、成功したものとしてカテゴリ別サマリーや設定画面に反映しない。

## 制約

- 初期表示件数は最大 3 件とする。
- ピン留め登録数は最大 3 件とする。
- 4 件目以降の表示は追加取得ではなく、取得済みデータの段階的な表示として扱う。
- ピン留め状態は `categories` 本体ではなく、カテゴリ表示設定として扱う。
- ピン留め登録数の上限は将来増減する可能性があるため、DB constraint / trigger で固定しない。
- Book 所有モデル移行後に、カテゴリ表示設定も Book 単位の表示設定へ移行する前提を壊さない。
- 既存の支払い一覧、カテゴリ作成、カテゴリ編集、カテゴリ削除、カテゴリ別サマリーの主要挙動を壊さない。
- ピン留め状態は一時的な画面 state だけに保持せず、source of truth 側の更新と再取得で整合させる。
- 新規依存の追加はこの要求の前提にしない。
- 実装方式、DB 変更有無、API 仕様、UI 詳細は Design Doc で既存仕様を調査して決める。

- AC-17: 既存の支払い一覧、カテゴリ作成、カテゴリ編集、カテゴリ削除、カテゴリ別サマリーの主要挙動が維持されている。

## Q&Aログ

- Q: Issue #1215 は最終仕様として扱うか？
  - A: 扱わない。Issue は Requirements / PRD の入力として扱い、既存コード・関連 Issue・ドキュメントから要求と受け入れ条件へ展開する。
- Q: `Show more` 操作はサーバーから追加取得するページングか？
  - A: 違う。取得済みのカテゴリ別サマリーを画面上で段階的に表示する操作として扱う。
- Q: ピン留め状態は `categories` に持たせるか？
  - A: 持たせない。カテゴリ本体ではなく、カテゴリ表示設定として扱う。
- Q: ピン留めカテゴリが 4 件以上になる操作は許可するか？
  - A: 許可しない。ピン留め登録数は最大 3 件とし、4 件目のピン留め登録は成功扱いにしない。
- Q: ピン留めカテゴリが 3 件未満の場合、初期表示枠を空けるか？
  - A: 空けない。未ピン留めカテゴリをカテゴリ ID 昇順で補い、最大 3 件まで表示する。
- Q: カテゴリ作成・更新とピン留め状態変更は片方だけ成功してよいか？
  - A: よくない。カテゴリ作成とピン作成、カテゴリ更新とピン作成/削除は、片方だけ成功する状態を避ける。
- Q: ピン留め登録数の上限は DB constraint / trigger で固定するか？
  - A: 固定しない。今後増減する可能性があるため、Design Doc で固定的な DB constraint / trigger 以外の担保方法を整理する。
- Q: 未分類 `Unknown` 行は初期表示件数と追加表示の対象に含めるか？
  - A: Design Doc で、既存体験と受け入れ条件に照らして明確にする。
- Q: 関連 Sub-issues は PRD の入力に含めるか？
  - A: 含める。ただし Requirements / PRD では実装手順として固定しない。
- Q: 現行実装にある `category_pins` はカテゴリ表示設定と矛盾するか？
  - A: 矛盾とは扱わない。現行実装ではカテゴリ本体とは別のピン状態として存在するため、PRD では「カテゴリ表示設定として扱う」という要求に留め、具体的なテーブル名や保存方式は Design Doc で確認する。

## 技術的考慮事項

- 現行のカテゴリ別サマリーは全取得結果を表示し、表示数を最大 3 件に制限する挙動や `Show more` 操作はない。
- 現行のカテゴリ設定取得では、各カテゴリのピン状態を確認できる。
- 現行の設定画面ではピン留め済みカテゴリに `Pin` バッジを表示するが、作成・編集フォームでピン留め状態を変更する UI はない。
- 現行のピン留め状態は、ピン留め登録数の最大 3 件制限を持たない。
- Book 所有モデル移行後、カテゴリピンはカテゴリの Book 所属とユーザーの Book membership に基づく境界チェックを持つ。
- カテゴリ作成・名前変更・削除では、カテゴリ一覧やカテゴリ別サマリーを最新化する既存経路がある。ピン留め状態変更でも、設定画面とカテゴリ別サマリーが古い状態を表示し続けないように考慮する必要がある。
