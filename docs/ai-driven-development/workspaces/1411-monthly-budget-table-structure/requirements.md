# Requirements / PRD: 月予算テーブルの構造を「予算なし」状態に対応させる

## 背景と課題

月次予算は、Book ごとに対象月の月初日を `effective_from` として保存されている。現行の有効月次予算は、対象月末日以前に開始した予算のうち `effective_from` が最新の 1 件として扱われる。

このため、月次予算を履歴のように積み上げている状態では、最新の予算レコードを削除すると、その前に登録された予算レコードが再び有効になる。ユーザーが最新予算を消したい、またはその月以降を「予算なし」にしたい意図で操作した場合でも、過去の予算が暗黙に復活し、画面上は予算が残っているように見える。

この要求は GitHub Issue #1411 を入力として整理する。Issue は最終仕様ではなく、Requirements / PRD のための意図、境界、成功条件として扱う。

## 対象ユーザーと利用シーン

- 対象ユーザー: 個人の支払い・貯蓄管理をこのアプリで行う認証済みユーザー。
- 利用シーン: 月次予算を設定、変更、削除または無効化し、支出状況の予算差分を確認する。
- 関連する現行体験:
  - 月予算がある場合、月次合計やサマリーで予算との差分を確認できる。
  - 月予算がない場合、予算差分は表示されない。
  - 月予算の作成では対象月と金額を入力し、対象月の月初日が `effective_from` として保存される。
  - 月予算の取得では、対象月末日以前の `effective_from` を降順に並べ、最新 1 件を有効予算として取得している。

## ユーザーストーリー

- 認証済みユーザーとして、最新の月予算を削除または無効化したい。そうすることで、その月以降を予算なしとして扱い、過去の予算が勝手に復活しない状態にできる。
- 認証済みユーザーとして、予算がない月は予算差分を表示しないでほしい。そうすることで、意図していない過去予算をもとに残額や超過額を判断せずに済む。
- 認証済みユーザーとして、当月に有効な月予算の金額を変更したい。そうすることで、過去月の予算履歴を壊さずに、当月以降の予算だけを見直せる。
- 認証済みユーザーとして、あとから新しい月予算を設定したい。そうすることで、「予算なし」にした期間のあとに、再び予算管理を始められる。

## スコープ

### 対象

- 月予算テーブルまたは関連する永続データ構造で、「予算なし」を明示的に表現できること。
- 有効な月予算の判定ルールで、「予算なし」が過去の予算レコードより優先されること。
- 最新の月予算を削除または無効化したあと、過去予算が暗黙に復活しないこと。
- 当月に有効な月予算の更新が、過去月の予算表示を上書きしないこと。
- 過去月に対する新規予算登録や更新を制限すること。
- 月予算がない状態の月次合計、サマリー、予算使用状況の期待挙動。
- 既存データを新しい構造や判定ルールへ移行するための要求整理。

### 対象外

- 月予算以外の予算管理構造の見直し。
- UI の大幅な再設計。
- 履歴表示や監査ログ機能の追加。
- 月予算の複数 Book 対応や Book 所有モデルの見直し。
- Requirements / PRD 作成 Goal 内での実装、Design Doc 作成、PR 作成。

## 機能要件

### FR-1: 「予算なし」を明示的に扱える

- ユーザーが月予算を削除または無効化した結果として、「予算なし」の状態を永続的に表現できる。
- 「予算なし」は単に最新予算レコードが存在しない状態とは区別できる。
- 「予算なし」は、同じ Book の過去の月予算より優先して有効状態を決める。
- この Requirements / PRD では、具体的な表現方法を固定しない。例として、イベント、状態種別、終端日、別テーブル、RPC などは Design Doc で比較する。

### FR-2: 最新月予算を削除または無効化しても過去予算が復活しない

- 削除または無効化できる月予算は、現在有効な最新月予算に限る。
- 現在有効な最新月予算を削除または無効化した場合、操作月以降は「予算なし」として扱われる。
- 操作月より前の対象月については、削除または無効化した月予算の履歴を閲覧できる。
- 「予算なし」のあとに新しい月予算を設定した場合、新しい月予算の対象月以降はその予算を有効として扱える。
- 過去予算が復活するかどうかは、物理的なレコード削除結果ではなく、ユーザーが選んだ予算状態に基づいて決まる。

### FR-3: 月予算の登録と更新は当月以降に限定する

- ユーザーは操作月以降を対象に、新しい月予算を登録できる。
- ユーザーは操作月より前の月を対象に、新しい月予算を登録できない。
- 更新できるのは、操作月に有効な月予算のみである。
- 操作月に有効な月予算を更新した場合、過去月に表示される予算額は変更しない。
- 操作月に有効な月予算の更新結果は、操作月以降の有効予算として扱う。
- この Requirements / PRD では、更新を既存レコードの変更として扱うか、新しい有効状態の追加として扱うかは固定しない。ただし、過去月の予算表示を上書きしてはいけない。

### FR-4: 有効月予算の判定が「金額あり」と「予算なし」を区別する

- 対象月の有効状態は、月予算金額がある状態、予算なし状態、未設定状態を区別して判定できる。
- 月予算金額がある状態では、既存どおり予算額と月次支出合計から残額または超過額を計算する。
- 予算なし状態では、月次支出合計は表示できるが、予算差分は表示しない。
- 未設定状態と予算なし状態のユーザー向け表示を同じにするか分けるかは、Design Doc で既存 UI と照らして決める。ただし、内部判定では過去予算の復活を防げる必要がある。

### FR-5: 既存データの意味を壊さず移行できる

- 既存の月予算レコードは、移行後も金額ありの月予算として扱える。
- 移行直後に、既存ユーザーの有効月予算が意図せず「予算なし」へ変わらない。
- 既存データから復元できないユーザーの削除意図は、移行で推測しない。
- 移行後に「予算なし」操作を行った時点から、新しい判定ルールに基づいて過去予算の復活を防ぐ。

### FR-6: 失敗時にユーザーが失敗を認識できる

- 月予算の削除または無効化が失敗した場合、ユーザーは画面上で失敗を認識できる。
- 失敗した操作を、予算なしになったものとして扱わない。
- 月予算状態の保存に失敗した場合、月次合計やサマリーに成功後の状態を先取りして表示し続けない。

## 代表シナリオ

2026/10/01 時点で、次の月予算が存在する場合:

- 予算01: 100,000円、2026/04 から有効
- 予算02: 120,000円、2026/08 から有効
- 予算03: 90,000円、2026/09 から有効

期待する挙動:

- 削除または無効化できるのは、現在有効な最新予算である予算03だけである。
- 予算03を削除または無効化した場合:
  - 2026/10 の予算はなしになる。
  - 2026/09 の支払い一覧では、予算03の 90,000円を月予算として閲覧できる。
  - 予算02の 120,000円は、2026/10 の有効予算として復活しない。
- 予算03を削除または無効化したあと、予算04として 100,000円を 2026/10 から登録できる。
- 2026/10/01 時点では、2026/09 から有効な新しい月予算は登録できない。
- 予算03を 100,000円に更新した場合:
  - 2026/10 の予算は 100,000円になる。
  - 2026/09 の支払い一覧では、予算03の更新前金額である 90,000円を月予算として閲覧できる。
  - 更新操作によって、過去月である 2026/09 の予算履歴を上書きしない。

## 非機能要件と制約

- 月予算は Book に属するという既存境界を維持する。
- 金額は既存の金額ドメインルールに従い、金額ありの月予算では 0 以上の整数として扱う。
- 「予算なし」を金額 0 と同一視しない。0 円予算と予算なしは異なる状態として扱える必要がある。
- 既存の月次支出合計、支払い一覧、カテゴリ関連機能の主要挙動を壊さない。
- 新規依存の追加はこの要求の前提にしない。
- DB 変更や API 仕様変更が必要な場合、Design Doc で影響範囲、移行方針、互換性、検証方針を明確にする。
- 参考リンクは設計思想の参考として扱い、特定記事の実装案をこの PRD の結論として固定しない。

## 受け入れ条件

- AC-1: 最新の月予算を削除または無効化しても、前の月予算が有効月予算として復活しない。
- AC-2: 現在有効な最新月予算を削除または無効化した操作月以降は、次の月予算が設定されるまで「予算なし」として扱われる。
- AC-3: 現在有効な最新月予算を削除または無効化しても、操作月より前の対象月ではその月に有効だった予算を閲覧できる。
- AC-4: 「予算なし」の月では、月次支出合計は確認できるが、予算残額または超過額は表示されない。
- AC-5: 「予算なし」のあとに新しい月予算を設定すると、その対象月以降は新しい予算額が有効になる。
- AC-6: 0 円予算と予算なしを区別できる。
- AC-7: 削除または無効化できる月予算は、現在有効な最新月予算に限られる。
- AC-8: 操作月より前の月を対象に、新しい月予算を登録できない。
- AC-9: 操作月に有効な月予算を更新した場合、操作月以降の予算額だけが変わり、過去月の予算表示は変わらない。
- AC-10: 既存の月予算レコードは、移行後も金額ありの月予算として扱われる。
- AC-11: 移行直後に、既存ユーザーの有効月予算が意図せず変わらない。
- AC-12: 月予算状態の保存に失敗した場合、ユーザーに失敗が分かり、成功扱いにならない。
- AC-13: 既存の月次合計、支払い一覧、カテゴリ関連機能の主要挙動が維持されている。

## Q&Aログ

- Q: Issue #1411 は最終仕様として扱うか？
  - A: 扱わない。Issue は Requirements / PRD の入力として扱い、既存コード・既存ドキュメントから要求と受け入れ条件へ展開する。
- Q: 「削除」は物理削除として固定するか？
  - A: 固定しない。ユーザー意図としては「操作月以降を予算なしにする操作」として整理し、物理削除、無効化、イベント追加などの実装方式は Design Doc で決める。
- Q: 最新月予算を削除または無効化した場合、どの期間を予算なしにするか？
  - A: 削除または無効化した操作月以降を予算なしとして扱い、次の月予算が設定されるまで過去予算を復活させない。
- Q: 削除前の過去月にも影響させるか？
  - A: 影響させない。操作月より前の対象月で有効だった過去予算は、不必要に失わせない。
- Q: 削除または無効化できる月予算はどれか？
  - A: 現在有効な最新月予算だけを対象にする。
- Q: 過去月に新しい月予算を登録できるか？
  - A: 登録できない。操作月より前の月を対象にした新規登録は許可しない。
- Q: 当月に有効な月予算を更新した場合、過去月の表示も変わるか？
  - A: 変わらない。更新は操作月以降の有効予算にだけ反映し、過去月で閲覧できる予算額は維持する。
- Q: 「予算なし」は金額 0 と同じか？
  - A: 同じではない。0 円予算は予算額がある状態、予算なしは予算差分を計算しない状態として区別する。
- Q: 既存データから過去の削除意図を復元するか？
  - A: 復元しない。過去に記録していないユーザーの削除意図は、移行で推測しない。
- Q: 参考記事の設計をそのまま採用するか？
  - A: 採用を固定しない。状態カラムや NULL の追加だけで複雑化するリスク、出来事を記録する考え方を Design Doc の比較材料にする。

## 技術的考慮事項

- 現行のドメイン文書では、有効な月次予算は「対象月末日以前に開始した予算のうち最新の 1 件」と定義されている。
- 現行の `fetchEffectiveMonthlyBudget` は `monthly_budgets` から `effective_from <= 対象月末日`、`effective_from desc`、`limit 1` で 1 件を取得している。
- 現行の `monthly_budgets` は `book_id, effective_year, effective_month` の一意制約を持ち、同一 Book・同一年・同月の月次予算を重複できない。
- 現行の月予算作成は `effective_from` と `amount` を insert し、月予算更新は対象レコードの `amount` を update する。
- 現行の Web 実装には月予算削除処理は見当たらない。Issue の「削除」は、今後追加または整理される操作意図として扱う必要がある。
- 現行の RLS には月予算の select / insert / update policy があるが、delete policy は見当たらない。
- 「予算なし」を表現する場合、既存の一意制約、読み取りクエリ、一覧表示、作成・更新フォーム、型定義、テストデータへの影響を Design Doc で確認する必要がある。
- 参考記事では、状態カラムや NULL 許容カラムを継ぎ足すだけでは要求追加に伴ってテーブルが複雑化しやすく、出来事を別の記録として扱う設計が比較対象として示されている。
- 参考 X 投稿は現在の取得環境では本文を確認できなかったため、Design Doc で必要なら監督者に内容確認を依頼する。
