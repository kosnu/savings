# Requirements / PRD: 設定画面でカテゴリを削除できるようにする

## 背景と課題

設定画面ではカテゴリの作成と名前変更ができるが、不要になったカテゴリを削除して整理する手段がない。

カテゴリが増えたまま整理できないと、支払い入力や支払い絞り込み、設定画面の確認時に不要な選択肢が残り続ける。ユーザーは現在使っているカテゴリだけを保ちたいが、不要カテゴリを取り除けないため、日常的な支払い管理の操作効率が下がる。

この要求は GitHub Issue #1279 を入力として整理する。Issue は最終仕様ではなく、Requirements / PRD のための意図、境界、成功条件として扱う。

## 対象ユーザーと利用シーン

- 対象ユーザー: 個人の支払い・貯蓄管理をこのアプリで行う認証済みユーザー。
- 利用シーン: 設定画面でカテゴリ一覧を見直し、不要になったカテゴリを削除して、以後の支払い入力や絞り込みに不要カテゴリが出ないようにする。
- 関連する現行体験:
  - 設定画面の Book 設定内でカテゴリ一覧を確認できる。
  - カテゴリは作成と名前変更ができる。
  - 支払い一覧、支払い詳細、支払い作成・更新、カテゴリフィルタでカテゴリ情報が使われている。

## ユーザーストーリー

- 認証済みユーザーとして、設定画面から不要なカテゴリを削除したい。そうすることで、今後の支払い入力やカテゴリ選択で不要なカテゴリを見なくてよくなる。
- 認証済みユーザーとして、過去の支払いに使っていたカテゴリでも削除したい。そうすることで、カテゴリ整理のために支払い履歴を先に手作業で修正しなくてよくなる。
- 認証済みユーザーとして、カテゴリを削除しても支払い自体は残っていてほしい。そうすることで、カテゴリ整理によって支払い履歴や金額の記録を失わずに済む。

## スコープ

### 対象

- 設定画面のカテゴリ削除体験。
- カテゴリ削除後に、削除されたカテゴリがカテゴリ一覧やカテゴリ選択肢に表示されないこと。
- カテゴリ削除後に、支払い情報から削除済みカテゴリ情報を参照できないこと。
- 支払いで使われているカテゴリでも、事前チェックなしに削除できること。
- カテゴリ削除に必要な最小限のデータ整合性要求。
- 初期実装としての同期削除を前提にした要求整理。

### 対象外

- 非同期削除。
- タグ/ラベル化。
- カテゴリの多対多化。
- タグ別予算、タグ別集計。
- 支払いで使われているカテゴリの削除前警告や事前チェック。
- 削除済みの予算履歴機能に関する対応。
- Requirements / PRD 作成Goal内での実装、Design Doc作成、PR作成。

## 機能要件

### FR-1: 設定画面からカテゴリを削除できる

- ユーザーは設定画面のカテゴリ一覧から、任意のカテゴリに対して削除操作を開始できる。
- 削除操作の対象は、現在ユーザーが操作できる Book に属するカテゴリに限る。
- 削除が成功したカテゴリは、設定画面のカテゴリ一覧に表示されない。

### FR-2: 削除済みカテゴリは今後のカテゴリ選択肢に表示されない

- 削除されたカテゴリは、支払い作成・支払い更新・支払い一覧のカテゴリフィルタなど、カテゴリ一覧をもとにした選択肢に表示されない。
- 削除後のカテゴリ一覧取得は、削除済みカテゴリを含まない最新状態を返す。

### FR-3: 支払いで使われているカテゴリでも事前チェックなしに削除できる

- 削除対象カテゴリが既存支払いで使われていても、削除前に利用有無チェックを要求しない。
- 削除対象カテゴリが支払いで使われていることを理由に、ユーザー操作をブロックしない。
- 支払いで使われているカテゴリの削除前警告や影響件数表示は初期スコープに含めない。

### FR-4: 削除後も支払い自体は残る

- カテゴリ削除によって、削除対象カテゴリを使っていた支払い自体は削除されない。
- 支払いの金額、日付、メモ、Book 所属など、カテゴリ以外の支払い情報は失われない。
- 削除済みカテゴリを使っていた支払いは、支払い一覧や支払い詳細で表示できる。

### FR-5: 支払い情報から削除済みカテゴリ情報を参照できない

- カテゴリ削除後、支払い一覧や支払い詳細から削除済みカテゴリの名前やIDを有効なカテゴリ情報として参照できない。
- 削除済みカテゴリを使っていた支払いは、カテゴリ未設定またはカテゴリ不明として扱われる。
- このRequirements / PRDでは、`payments.category_id` を `null` にするなどの具体的な実装方式は固定しない。

### FR-6: 削除失敗時にユーザーが失敗を認識できる

- カテゴリ削除が失敗した場合、ユーザーは画面上で失敗を認識できる。
- 削除失敗時、対象カテゴリは削除されたものとして扱わない。
- 削除失敗時、支払い情報が中途半端に欠落した状態を成功扱いしない。

## 非機能要件と制約

- 初期実装は同期削除を前提にする。
- 将来の非同期削除に拡張できる余地を塞がない。
- 新規依存の追加はこの要求の前提にしない。
- 既存の支払い一覧、支払い詳細、支払い作成・更新、カテゴリフィルタ、カテゴリ作成、カテゴリ名更新の主要挙動を壊さない。
- Book 所属に基づく既存の参照・操作境界と矛盾しない。
- 削除済みの予算履歴機能に関する挙動は要求に含めない。

## 受け入れ条件

- AC-1: 設定画面からカテゴリを削除できる。
- AC-2: 削除したカテゴリは、設定画面のカテゴリ一覧に表示されない。
- AC-3: 削除したカテゴリは、支払い作成・支払い更新・支払い一覧フィルタのカテゴリ選択肢に表示されない。
- AC-4: 支払いで使われているカテゴリでも、事前チェックなしに削除できる。
- AC-5: 削除したカテゴリを使っていた支払いは、削除後も支払い一覧に表示される。
- AC-6: 削除したカテゴリを使っていた支払いは、削除後も支払い詳細で表示できる。
- AC-7: 削除したカテゴリを使っていた支払いから、削除済みカテゴリの名前やIDを有効なカテゴリ情報として参照できない。
- AC-8: カテゴリ削除に失敗した場合、ユーザーに失敗が分かり、対象カテゴリは削除成功扱いにならない。
- AC-9: 削除済みの予算履歴機能に関する対応や表示が追加されていない。
- AC-10: 既存の支払い、カテゴリ一覧、カテゴリ作成、カテゴリ名更新の主要挙動が維持されている。

## Q&Aログ

- Q: Sub-issues は必要か？
  - A: 今すぐは不要。#1279 を Requirements / PRD の起点として使い、PRD / Design Doc 後に必要なら分割する。
- Q: 既存のカテゴリ削除 Sub-issues はどう扱うか？
  - A: #1279 の Sub-issues #1289, #1290, #1291, #1292 はすべて Close as not planned 済み。古い分割や古い前提を採用しない。
- Q: カテゴリに紐づく予算履歴は扱うか？
  - A: 扱わない。予算履歴機能は削除済みのため、このIssueの内容から除外する。
- Q: 支払い側の扱いとして `null` 化などの具体的な方式を要求に含めるか？
  - A: 含めない。「支払い情報から削除したカテゴリ情報が参照できないこと」という要求として扱う。
- Q: 支払いで使われているかどうかの事前チェックは必要か？
  - A: 必要ない。初期スコープでは事前チェックなしに削除できることを要求する。

## 技術的考慮事項

- 現行DB定義では、`payments.category_id` はカテゴリへの nullable な参照として扱われている。
- 現行の支払い一覧・詳細取得は、支払いからカテゴリを JOIN してカテゴリ名を表示している。
- 現行の支払い一覧・詳細UIは、カテゴリ情報が取得できない場合に `Unknown` 表示へ落ちる経路を持つ。
- 現行のカテゴリ設定画面はカテゴリ作成と名前変更を持つが、削除操作はない。
- 現行のカテゴリ設定取得は `categories` を起点にし、カテゴリピン状態も併せて表示している。カテゴリ削除時はピン情報が削除済みカテゴリを残さないこともデータ整合性として考慮する。
- 現行の `categories` RLS には read / insert / update の方針があるが、削除操作は別途確認が必要。
- 予算履歴に関する古い migration や関数は残っているが、現在は削除 migration により `category_budgets` は対象外として扱う。
- Design Doc では、削除後の支払い表示を「カテゴリ未設定」と同じ扱いにするか、「Unknown」として扱うかを既存UIとの整合性から決める必要がある。ただし、この PRD では削除済みカテゴリ情報を参照できないことを要求として固定し、具体的な保存方式は固定しない。
