# Requirements / PRD: カテゴリ別予算機能を追加する

## 背景と課題

設定画面では、カテゴリの作成、名前更新、ピン留め、削除を扱える。カテゴリは Book に属し、削除されたカテゴリに紐づく支払いは未分類として扱われる。

一方で、現在の有効な Web UI では、カテゴリ作成や更新の操作でカテゴリ別予算を扱えない。カテゴリ一覧もカテゴリ名とピン留め状態だけを表示し、支払い一覧サマリーもカテゴリ名と対象期間の支払い合計だけを表示する。過去の migration にはカテゴリ別予算の痕跡があるが、後続 migration で削除されているため、現在の有効機能としては扱わない。

ユーザーはカテゴリを管理する文脈でカテゴリ別予算も管理し、カテゴリ一覧で設定状況を確認したい。また、支払い一覧サマリーではカテゴリ別の合計額が予算に対してどういう状態かを判断したい。スマホでは、カテゴリサマリーを1列の読み順で確認したい。

## 対象ユーザーと利用シーン

- 対象ユーザー: savings の Web UI でカテゴリ、支払い、予算状況を管理する認証済みユーザー。
- カテゴリ作成時: 新しいカテゴリに、必要なら予算も同じ管理操作の中で設定する。
- カテゴリ更新時: 既存カテゴリの意味や使い方に合わせて、予算を設定、変更、解除する。
- カテゴリ一覧確認時: 各カテゴリに予算があるか、ある場合はいくらかを確認する。
- 支払い一覧サマリー確認時: 対象月のカテゴリ別支払い合計と、カテゴリ別予算に対する差を判断する。
- スマホ利用時: 狭い画面でもカテゴリ別サマリーを1列で読み、カテゴリ、合計額、予算との差の関係を追う。

## ユーザーストーリー

- 認証済みユーザーとして、カテゴリ作成時にカテゴリ別予算も一緒に設定したい。そうすることで、カテゴリ作成後に別の場所で予算を探して設定し直さずに済む。
- 認証済みユーザーとして、カテゴリ更新時にカテゴリ別予算を設定、変更、解除したい。そうすることで、カテゴリの使い方が変わったときに基準値も同じ文脈で見直せる。
- 認証済みユーザーとして、カテゴリ削除後は、そのカテゴリの予算が通常の有効予算として後続画面に残らないでほしい。そうすることで、削除済みカテゴリを現在使えるカテゴリとして誤認しない。
- 認証済みユーザーとして、カテゴリ一覧で各カテゴリの予算を確認したい。そうすることで、設定済みの基準値を管理画面で把握できる。
- 認証済みユーザーとして、支払い一覧サマリーでカテゴリ別合計と予算との差を確認したい。そうすることで、カテゴリごとの支出状況を予算に対して判断できる。

## スコープ

### 対象

- カテゴリ作成時に、カテゴリ別予算を同じ操作文脈で登録できること。
- カテゴリ更新時に、カテゴリ別予算を設定、変更、解除できること。
- カテゴリ削除後、そのカテゴリの予算が通常の有効予算としてカテゴリ一覧や支払い一覧サマリーに残らないこと。
- カテゴリ一覧で、カテゴリ別予算の有無と金額を確認できること。
- 支払い一覧サマリーで、カテゴリ別の対象月合計額と予算との差を確認できること。
- スマホ幅のカテゴリサマリーを1列の読み順にすること。

### 対象外

- 月次予算や全体予算の仕様見直し。
- カテゴリ以外を対象にした予算機能の追加。
- 支払い登録・更新フローへの予算入力の追加。
- 具体的な DB schema、RPC 名、API 形状、migration 方針の決定。
- 具体的な component、列構成、breakpoint、最終 copy、テスト実装方針の決定。

## 機能要件

### FR-1: カテゴリ作成時に予算を一緒に登録できる

- ユーザーはカテゴリ作成時に、カテゴリ名や既存の表示設定と同じ操作文脈でカテゴリ別予算を登録できる。
- 予算を入力しないカテゴリ作成もできる。
- 予算未入力は、0 円予算ではなく予算未設定として扱う。
- カテゴリ作成と予算登録が1つの操作として見える場合、片方だけ成功して完了済みに見えてはいけない。

### FR-2: カテゴリ更新時に予算を設定、変更、解除できる

- ユーザーは既存カテゴリにカテゴリ別予算を設定できる。
- ユーザーは設定済みのカテゴリ別予算額を変更できる。
- ユーザーはカテゴリ別予算を解除または未設定化できる。
- 解除や未設定化は、0 円予算の設定と同じ意味にしてはいけない。

### FR-3: カテゴリ削除後に予算が有効表示として残らない

- カテゴリ削除後、そのカテゴリの予算はカテゴリ一覧や支払い一覧サマリーで通常の有効予算として表示されない。
- 削除済みカテゴリの扱いは、既存の支払い未分類化ルールと矛盾しない。
- 物理削除、状態変更、履歴保持などの方式は Requirements では固定しない。

### FR-4: カテゴリ一覧で予算を確認できる

- ユーザーはカテゴリ一覧で、各カテゴリに予算が設定されているかを確認できる。
- 予算が設定されている場合、ユーザーは金額を確認できる。
- 予算がない状態は、0 円予算や取得失敗と混同しない。
- 具体的な列構成、表示順、component、文言は Design で決める。

### FR-5: 支払い一覧サマリーで予算との差を確認できる

- ユーザーは支払い一覧サマリーで、カテゴリ別の対象月合計額を確認できる。
- 予算があるカテゴリでは、対象月合計額とカテゴリ別予算との差を確認できる。
- 差は、予算に対する残り、超過、一致などを判断できる比較結果として扱う。
- 予算未設定、0 円予算、計算不能、取得失敗を混同しない。

### FR-6: スマホのカテゴリサマリーを1列にする

- スマホ幅では、支払い一覧サマリーのカテゴリ表示を1列の読み順で確認できる。
- 1列表示でも、カテゴリ、月合計額、予算との差の関係が分かる。
- PC 幅の列数や breakpoint は Requirements では固定しない。

### FR-7: 失敗時に完了済みに見せない

- カテゴリ作成、更新、削除、予算保存、予算取得が失敗した場合、ユーザーは失敗を認識できる。
- 失敗した保存や削除を、成功後の予算状態として扱わない。
- 取得失敗や計算不能を、予算なしや 0 差として表示しない。

## 期間・履歴・削除の未決事項

Issue #1440 だけでは、カテゴリ別予算を現在値として扱うか、月別状態や履歴を持つ値として扱うかは確定しない。Design / Plan では、Temporal Data Policy に沿って少なくとも次を決める。

- 予算が適用される基準月。
- 予算変更が過去月、現在月、未来月に与える影響。
- 予算解除後に過去の予算が暗黙に復活しないことをどう保証するか。
- カテゴリ削除後、過去月表示と現在有効表示をどう扱うか。
- 登録、更新、解除を許可する対象月や対象期間。

## 非機能要件と制約

- カテゴリは Book に属する既存境界を維持する。
- カテゴリ名、ピン留め、支払いとの関係は既存カテゴリルールを壊さない。
- 金額は既存金額ルールに従い、0 以上の整数として扱う。
- カテゴリ別予算は月次予算や全体予算の仕様見直しへ広げない。
- 支払い登録・更新フローに予算入力を追加しない。
- 新規依存はこの Requirements の前提にしない。
- DB/API/RLS/RPC/migration が必要な場合は、Design / Plan で影響範囲、操作単位、検証方針を整理する。

## Q&Aログ

- Q: 過去に作成された requirements、design、code を参考にするか？
  - A: しない。今回の PRD は Issue #1440、監督入力、選択 docs、現在の有効実装から作り直す。
- Q: カテゴリ別予算は現在ある機能として扱うか？
  - A: 扱わない。過去 migration の痕跡はあるが、削除 migration 後の現在有効機能ではない。
- Q: カテゴリ一覧やサマリーの具体的な layout を Requirements で決めるか？
  - A: 決めない。ユーザーが判断したい値と状態を定義し、列構成、component、copy は Design で決める。
- Q: カテゴリ別予算は月別履歴を持つ前提か？
  - A: Issue だけでは確定しない。対象月との比較が必要なため、Design で期間、履歴、解除、削除の扱いを決める。
- Q: 「紐づく予算も削除」は物理削除を意味するか？
  - A: Requirements では固定しない。ユーザーにとっては削除済みカテゴリの予算が通常の有効予算として残らないことが目的である。

## 技術的考慮事項

- 現行のカテゴリ作成/更新は、カテゴリ名とピン留めを同じ操作単位で扱う。
- 現行のカテゴリ削除はカテゴリを削除し、支払いは未分類、ピン留めは削除される。
- 現行のカテゴリ一覧はカテゴリ名とピン留め状態を表示し、予算は表示しない。
- 現行の支払い一覧サマリーはカテゴリ名、対象期間の支払い合計、ピン留め状態、未分類合計を扱い、カテゴリ別予算や差は扱わない。
- 現行のカテゴリサマリーは複数列で表示され、スマホ専用の1列 requirement は未整理である。
- Design / Plan では、カテゴリ作成/更新/削除と予算変更をどの操作単位で閉じるかを決める必要がある。
- Design / Plan では、予算の対象月、履歴、解除、削除済みカテゴリ、取得失敗の扱いを決める必要がある。
