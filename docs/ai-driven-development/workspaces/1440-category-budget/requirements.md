# Requirements / PRD: カテゴリ別予算機能を追加する

## 背景と課題

設定画面のカテゴリ管理では、カテゴリ名やピン留め状態を作成、更新、削除できる。一方で、カテゴリごとの予算は現在の有効機能として扱われていない。過去の migration には `category_budgets` とカテゴリ作成・更新に予算を含める関数の痕跡があるが、現在は `20260528000000_remove_category_budgets.sql` で削除されているため、この PRD では future capability として扱う。

ユーザーはカテゴリ管理の文脈でカテゴリ別予算を登録・更新し、カテゴリ一覧で各カテゴリの予算を確認したい。また、支払い一覧の月次サマリーでは、カテゴリごとの月合計額とカテゴリ別予算との差分を確認したい。スマートフォンでは、現在 2 列で表示しているカテゴリ別サマリーを 1 列にし、狭い画面でもカテゴリごとの情報を読みやすくしたい。

この要求は GitHub Issue #1440 を入力として整理する。Issue は最終仕様ではなく、Requirements / PRD のための意図、境界、成功条件として扱う。

## 対象ユーザーと利用シーン

- 対象ユーザー: savings の Web UI でカテゴリ、支払い一覧、月次サマリーを管理する認証済みユーザー。
- 設定画面での利用シーン: カテゴリを作成または更新するときに、同じカテゴリ管理操作の中でカテゴリ別予算も扱う。
- カテゴリ一覧での利用シーン: 登録済みカテゴリを確認しながら、カテゴリに予算が設定されているか、設定されている場合はいくらかを確認する。
- 支払い一覧での利用シーン: 対象月のカテゴリ別支払い合計を見ながら、そのカテゴリの予算に対して残っているのか超過しているのかを判断する。
- モバイルでの利用シーン: 支払い一覧サマリーのカテゴリ別情報を 1 列の読み順で確認する。

## ユーザーストーリー

- 認証済みユーザーとして、カテゴリ作成時にカテゴリ別予算を一緒に登録したい。そうすることで、カテゴリ管理と予算管理を別々に行わずに済む。
- 認証済みユーザーとして、カテゴリ更新時にカテゴリ別予算を登録、変更、削除したい。そうすることで、カテゴリの意味や使い方が変わったときに予算も同じ管理文脈で見直せる。
- 認証済みユーザーとして、カテゴリ削除時に紐づくカテゴリ別予算も削除または無効化されてほしい。そうすることで、削除済みカテゴリの予算が後続の一覧や集計に残って見えない。
- 認証済みユーザーとして、カテゴリ一覧でカテゴリ別予算を確認したい。そうすることで、どのカテゴリに予算が設定されているかを設定画面で判断できる。
- 認証済みユーザーとして、支払い一覧サマリーでカテゴリ名、月合計額、予算との差分を確認したい。そうすることで、カテゴリごとの支出状況を予算に対して判断できる。
- 認証済みユーザーとして、スマートフォンではカテゴリ別サマリーを 1 列で読みたい。そうすることで、狭い画面でもカテゴリ名、月合計額、差分を追いやすくなる。

## スコープ

### 対象

- カテゴリ作成時に、カテゴリ別予算を同じユーザー操作の中で登録できること。
- カテゴリ更新時に、カテゴリ別予算を登録、更新、削除できること。
- カテゴリ削除時に、紐づくカテゴリ別予算が後続の一覧や集計に残らないこと。
- カテゴリ一覧で、カテゴリ別予算を確認できること。
- 支払い一覧のサマリーで、カテゴリ別の月合計額とカテゴリ別予算との差分を確認できること。
- スマートフォンの支払い一覧サマリーで、カテゴリ表示を 1 列にすること。
- カテゴリ別予算に関わる金額、期間、差分、未設定、削除後状態の要求整理。

### 対象外

- 月次予算や全体予算の仕様見直し。
- カテゴリ以外を対象にした予算機能の追加。
- 支払い登録・更新フローへの予算入力の追加。
- カテゴリ別予算の具体的な DB schema、RPC、API、migration 方針の決定。
- カテゴリ一覧や支払いサマリーの具体的な layout、表示順、component、最終 copy の決定。
- Requirements / PRD 作成 Goal 内での実装、Design Doc 作成、PR 作成。

## Domain Value Intent

UI に表示、入力、比較、集計、状態化する値は、次の判断目的を持つ。

| 値 | 利用目的 | Requirements 上の境界 |
| --- | --- | --- |
| カテゴリ | 対象を識別したい | カテゴリ名や未分類の扱いは既存カテゴリドメインと整合させる。表示順や component は固定しない。 |
| カテゴリ別予算額 | 実値を知りたい / 基準値を管理したい | 金額は 0 以上の整数として扱う。予算未設定と 0 円予算を混同しない。 |
| 月合計額 | 集計値を知りたい | 対象月の支払い合計として扱う。カテゴリ別予算の存在によって支払い合計そのものの意味を変えない。 |
| 差分 | 比較したい / 残量や超過を知りたい | 予算額と月合計額の比較結果として扱う。予算未設定、計算不能、取得失敗を 0 差分として表示しない。 |
| 月 / 期間 | 比較対象の期間を理解したい | 支払い一覧サマリーの対象月を基準にする。過去月、現在月、未来月に対する予算変更可否は Design / Plan で決める。 |
| 削除後状態 | 対象が後続表示に残るか知りたい | 削除、無効化、未設定化、0 円設定を混同しない。履歴や過去月表示が必要な場合は Design / Plan で決める。 |

## 機能要件

### FR-1: カテゴリ作成時にカテゴリ別予算を登録できる

- ユーザーはカテゴリ作成時に、カテゴリ名と同じ操作文脈でカテゴリ別予算額を登録できる。
- カテゴリ別予算を入力しないカテゴリ作成も可能である。
- カテゴリ作成とカテゴリ別予算登録は、ユーザーには 1 回の作成操作として理解できる。
- 片方だけ成功すると作成結果が誤解される場合、同一操作単位として扱う必要がある。
- この Requirements / PRD では、同一操作単位を DB transaction、RPC、API、アプリケーション層のどこで作るかは固定しない。

### FR-2: カテゴリ更新時にカテゴリ別予算を登録、更新、削除できる

- ユーザーは既存カテゴリの更新時に、カテゴリ別予算を新しく登録できる。
- ユーザーは既存カテゴリの更新時に、設定済みカテゴリ別予算額を変更できる。
- ユーザーは既存カテゴリの更新時に、設定済みカテゴリ別予算を削除または未設定化できる。
- 予算を削除または未設定化する操作は、0 円予算の登録と同じ意味にしてはいけない。
- 期間や履歴を持つカテゴリ別予算として扱う場合、更新・削除が過去月の表示を変えるかどうかを Design / Plan で決める。

### FR-3: カテゴリ削除時に紐づくカテゴリ別予算が後続表示に残らない

- ユーザーがカテゴリを削除した場合、そのカテゴリに紐づくカテゴリ別予算は、カテゴリ一覧や支払い一覧サマリーで通常の有効予算として表示されない。
- カテゴリ削除後、削除済みカテゴリの予算が現在有効な設定として残って見えてはいけない。
- 物理削除、無効化、履歴保持、過去月参照のどれで表現するかは Requirements / PRD では固定しない。
- カテゴリ削除により過去の支払いが未分類になる既存カテゴリルールと、過去月のカテゴリ別予算表示をどう扱うかは Design / Plan で決める。

### FR-4: カテゴリ一覧でカテゴリ別予算を確認できる

- ユーザーはカテゴリ一覧で、各カテゴリにカテゴリ別予算が設定されているかを確認できる。
- カテゴリ別予算が設定されている場合、ユーザーは予算額を確認できる。
- カテゴリ別予算が設定されていない場合、予算額がある状態や 0 円予算と混同しない。
- ここでは「確認できる」ことを要求し、具体的な列構成、表示順、状態ラベル、component、文言は固定しない。

### FR-5: 支払い一覧サマリーでカテゴリ別の月合計額と予算との差分を確認できる

- ユーザーは支払い一覧サマリーで、カテゴリごとの月合計額を確認できる。
- ユーザーはカテゴリ別予算があるカテゴリについて、対象月の月合計額とカテゴリ別予算との差分を確認できる。
- 差分は単なる符号付き金額ではなく、何に対する差なのかを判断できる比較結果として扱う。
- 予算未設定、0 円予算、計算不能、取得失敗を混同してはいけない。
- 支払いサマリーの主目的は、カテゴリ名、月合計額、予算との差分を判断できることである。追加指標、詳細な内訳、固定 layout はこの PRD では要求しない。

### FR-6: スマートフォンのカテゴリ別サマリーを 1 列で表示する

- スマートフォン幅では、支払い一覧サマリーのカテゴリ表示を 1 列の読み順で確認できる。
- 1 列表示でも、カテゴリ識別、月合計額、予算との差分の関係が分かる。
- PC や広い画面での列数、breakpoint、具体的な responsive props は Design / Plan で決める。

### FR-7: 失敗時にユーザーが失敗を認識できる

- カテゴリ作成、更新、削除とカテゴリ別予算の保存が失敗した場合、ユーザーは失敗を認識できる。
- 失敗した保存や削除を、成功後のカテゴリ別予算状態として扱わない。
- 部分成功が起きる可能性がある場合、ユーザーに完了済みと誤認させない。

## 期間・履歴・削除の扱い

カテゴリ別予算は、支払い一覧サマリーの対象月と結びつく可能性が高い。ただし Issue #1440 だけでは、カテゴリ別予算を「現在値」として扱うのか、月次状態や履歴を持つ値として扱うのかは確定しない。

この PRD では、次を Requirements の境界として残す。

- 支払い一覧サマリーの差分は、対象月のカテゴリ別支払い合計と、その対象月に適用されるカテゴリ別予算を比較する。
- カテゴリ別予算の変更が過去月に反映されるか、現在月以降にだけ反映されるかは Design / Plan で決める。
- カテゴリ別予算の削除が、物理削除、予算なし状態、無効化、履歴終端のどれを意味するかは Design / Plan で決める。
- 過去月表示を壊してはいけない要求を採用する場合は、Temporal Data Policy に沿って基準月、現在有効状態、過去時点で表示する状態を Design / Plan で明示する。

## 非機能要件と制約

- カテゴリは Book に属するという既存境界を維持する。
- カテゴリ名、ピン留め、支払いとの関係は既存カテゴリドメインルールを壊さない。
- 金額は既存の金額ドメインルールに従い、0 以上の整数として扱う。
- 予算未設定、0 円予算、削除済み、取得失敗、計算不能を混同しない。
- カテゴリ別予算は月次予算や全体予算の仕様見直しに広げない。
- 支払い登録・更新フローに予算入力を追加しない。
- 新規依存の追加はこの要求の前提にしない。
- DB 変更、API 仕様変更、RPC 追加、migration が必要な場合は、Design / Plan で影響範囲、移行、操作単位、検証を明確にする。

## 技術的考慮事項

- 現行のカテゴリ設定画面にはカテゴリ作成、カテゴリ名更新、カテゴリ削除、ピン留め表示がある。
- 現行のカテゴリ削除では、カテゴリレコードを削除し、関連クエリを更新する。カテゴリドメインルールでは、関連支払いはカテゴリ未設定、関連ピン留めは削除される。
- 過去の migration には `category_budgets`、`create_category_with_budget`、`update_category_with_budget` があったが、現在は削除済みである。Design / Plan では過去設計を現行仕様として前提にしない。
- 現行の支払い一覧サマリーのカテゴリ別合計は、カテゴリ、支払い、ピン留めからカテゴリ名と月合計額を作る。カテゴリ別予算額や差分は現在の戻り値に含まれない。
- 現行のカテゴリ別サマリーは `initialVisibleCount = 3`、既定 `chunkSize = 2` で表示し、スマートフォン専用の 1 列 requirement は未実装である。
- Design / Plan では、カテゴリ作成・更新・削除とカテゴリ別予算変更を同一操作単位にするか、どの層で操作境界を閉じるかを検討する。
- Design / Plan では、カテゴリ別予算の取得対象月、過去月表示、現在値、予算なし状態、未設定状態、削除済みカテゴリの扱いを決める必要がある。
