# Requirements / PRD: カテゴリ別予算機能を追加する

## 背景と課題

設定画面のカテゴリ管理では、ユーザーはカテゴリの作成、名前更新、ピン留め、削除を扱える。カテゴリは Book に属し、カテゴリ削除時には関連する支払いが未分類になり、関連するピン留めも削除される。

現行の有効な Web UI では、カテゴリ作成や更新の操作でカテゴリ別予算を扱えない。カテゴリ一覧ではカテゴリ名とピン留め状態、行操作を確認できるが、各カテゴリの予算状況は確認できない。支払い一覧サマリーでは、対象期間のカテゴリ別支払い合計を確認できるが、その合計がカテゴリ別予算に対してどういう状態かは判断できない。

一方、API migration には過去のカテゴリ別予算関連テーブルや関数の履歴があるが、後続 migration で削除されているものもある。Requirements では、過去実装の形を現在の要求として採用しない。現在ユーザーが必要としている体験と、現行の有効 UI の不足から要求を定義する。

## Current State / Current Gaps

- 現行の有効 Web UI では、カテゴリ登録/更新フォームに予算入力がない。
- 現行のカテゴリ一覧では、カテゴリ別予算の有無や金額を確認できない。
- 現行の支払い一覧サマリーでは、カテゴリ別支払い合計は確認できるが、カテゴリ別予算との差分は確認できない。
- 現行の支払い一覧サマリーは、スマホ幅でカテゴリ表示を1列の読み順にする要求を満たす前提で整理されていない。
- 予算の期間、履歴、月別扱い、削除後の状態は Issue #1440 だけでは確定しない。

## 対象ユーザーと利用シーン

- 対象ユーザー: savings の Web UI で支払い、カテゴリ、予算状況を管理する認証済みユーザー。
- カテゴリ作成時: 新しいカテゴリを作るときに、必要なら予算も同じ管理文脈で設定したい。
- カテゴリ更新時: カテゴリ名や表示設定を見直すときに、そのカテゴリの予算も登録、更新、削除したい。
- カテゴリ削除時: 削除したカテゴリに紐づく予算が、現在使えるカテゴリの予算として残らないことを確認したい。
- カテゴリ一覧確認時: 各カテゴリに予算があるか、ある場合はいくらかを把握したい。
- 支払い一覧サマリー確認時: 対象月のカテゴリ別支払い合計とカテゴリ別予算との差分を見て、カテゴリごとの支出状況を判断したい。
- スマホ利用時: 支払い一覧サマリーのカテゴリ表示を、狭い画面でも1列の読み順で確認したい。

## ユーザーストーリー

- 認証済みユーザーとして、カテゴリ登録時にカテゴリ別予算も一緒に登録したい。そうすることで、カテゴリ作成後に別の管理導線を探さずに済む。
- 認証済みユーザーとして、カテゴリ更新時にカテゴリ別予算を登録、更新、削除したい。そうすることで、カテゴリの使い方に合わせて基準額を見直せる。
- 認証済みユーザーとして、カテゴリ削除後に紐づく予算が現在使えるカテゴリの予算として残らないようにしたい。そうすることで、削除済みカテゴリを有効な管理対象として誤認しない。
- 認証済みユーザーとして、カテゴリ一覧でカテゴリ別予算を確認したい。そうすることで、カテゴリごとの基準額を管理画面で把握できる。
- 認証済みユーザーとして、支払い一覧サマリーでカテゴリ別合計と予算との差分を確認したい。そうすることで、対象月の支出状況をカテゴリごとに判断できる。

## スコープ

### 対象

- カテゴリ登録時に、カテゴリ別予算を一緒に登録できること。
- カテゴリ更新時に、カテゴリ別予算を登録、更新、削除できること。
- カテゴリ削除時に、紐づくカテゴリ別予算が通常の有効表示から外れること。
- カテゴリ一覧でカテゴリ別予算を確認できること。
- 支払い一覧サマリーで、カテゴリ別合計額とカテゴリ別予算との差分を確認できること。
- スマホ幅の支払い一覧サマリーでは、カテゴリ表示を1列の読み順にすること。

### 対象外

- 月次予算や全体予算の仕様見直し。
- カテゴリ以外を対象にした予算機能の追加。
- 支払い登録、更新フローへの予算入力の追加。
- 具体的な DB schema、RPC 名、API 形状、migration 方針の決定。
- 具体的な component、列構成、breakpoint、最終 copy、テスト実装方針の決定。
- 過去のカテゴリ別予算実装を復元すること自体。

## Domain Value Intent

| 値 | ユーザーが判断したいこと | Requirements 上の境界 |
| --- | --- | --- |
| カテゴリ | どの支出分類を管理・比較しているか | Book 所有、名前制約、削除時の支払い未分類化、ピン留めは既存カテゴリルールに従う。 |
| カテゴリ別予算額 | そのカテゴリの基準額があるか、ある場合はいくらか | 金額は0以上の整数として扱う。未入力、削除、0円予算を混同しない。 |
| カテゴリ別支払い合計 | 対象月にそのカテゴリでいくら使ったか | 支払い合計として扱い、予算の有無で合計値の意味を変えない。 |
| 予算との差分 | 予算に対して残っているか、超過しているか | 基準額と支払い合計の比較結果として扱う。基準額なし、計算不能、取得失敗を0差として扱わない。 |
| 月 / 期間 | どの期間の支払い合計と予算を比べているか | 支払い一覧サマリーの対象月を基準にする。変更が過去月、現在月、未来月にどう効くかは Design で決める。 |
| 削除 / 予算削除 / 未入力 / 0円 | 後続画面にどう残るか、比較できるか | 削除、未入力、0円予算、取得失敗を同じ状態として見せない。 |

## 機能要件

### FR-1: カテゴリ登録時に予算を一緒に登録できる

- ユーザーはカテゴリ登録時に、カテゴリ名や既存の表示設定と同じ管理文脈でカテゴリ別予算を登録できる。
- 予算を入力しないカテゴリ登録もできる。
- 予算未入力は、0円予算として扱ってはいけない。
- カテゴリ登録と予算登録が1つの操作として見える場合、片方だけ成功した状態を完了済みに見せてはいけない。

### FR-2: カテゴリ更新時に予算を登録、更新、削除できる

- ユーザーは既存カテゴリにカテゴリ別予算を登録できる。
- ユーザーは既存のカテゴリ別予算額を更新できる。
- ユーザーはカテゴリ別予算を削除できる。
- 予算削除は、未入力や0円予算と同じ意味にしてはいけない。

### FR-3: カテゴリ削除時に紐づく予算が通常表示から外れる

- カテゴリ削除後、そのカテゴリに紐づく予算は、カテゴリ一覧や支払い一覧サマリーで通常の有効なカテゴリ別予算として表示されない。
- カテゴリ削除後も、既存カテゴリルールどおり関連支払いは未分類として扱う。
- 削除済みカテゴリや削除済み予算の履歴表示を追加するかどうかは、この Requirements では固定しない。

### FR-4: カテゴリ一覧でカテゴリ別予算を確認できる

- ユーザーはカテゴリ一覧で、各カテゴリに予算があるかを確認できる。
- 予算がある場合、ユーザーはその金額を確認できる。
- 予算がない、予算が削除された、0円予算である、取得に失敗した、という状態を混同しない。
- 具体的な列構成、表示順、文言、component は Design で決める。

### FR-5: 支払い一覧サマリーで予算との差分を確認できる

- ユーザーは支払い一覧サマリーで、カテゴリ別の対象月合計額を確認できる。
- カテゴリ別予算があるカテゴリでは、対象月合計額と予算との差分を確認できる。
- 差分は、残額、超過、一致などの判断につながる比較結果として扱う。
- 予算なし、予算削除、0円予算、計算不能、取得失敗を0差として扱わない。

### FR-6: スマホのカテゴリサマリーを1列で読める

- スマホ幅では、支払い一覧サマリーのカテゴリ表示を1列の読み順で確認できる。
- 1列表示でも、カテゴリ、対象月合計額、予算との差分の関係が分かる。
- PC 幅の列数や breakpoint は Requirements では固定しない。

### FR-7: 失敗時に完了済みに見せない

- カテゴリ登録、更新、削除、予算保存、予算取得が失敗した場合、ユーザーは失敗を認識できる。
- 保存や削除の失敗後に、UIだけが成功後のカテゴリ別予算状態に進んだように見えてはいけない。

## 期間・履歴・削除の判断点

Issue #1440 だけでは、カテゴリ別予算を現在値として扱うか、月別有効状態として扱うか、履歴を保持するかは確定しない。Design / Plan では、Temporal Data Policy に沿って少なくとも次を決める。

- カテゴリ別予算が適用される基準日または基準月。
- 登録、更新、削除を許可する対象月や対象期間。
- 現在の変更が過去月、現在月、未来月に与える影響。
- 予算削除後に古い予算が暗黙に復活しないことをどう保証するか。
- カテゴリ削除後、過去時点表示と現在有効表示をどう扱うか。
- 予算なし、未入力、削除済み、0円予算、取得失敗をどう区別するか。

## 非機能要件と制約

- カテゴリは Book に属する既存境界を維持する。
- カテゴリ名、ピン留め、支払いとの関係は既存カテゴリルールを壊さない。
- 金額は既存金額ルールに従い、0以上の整数として扱う。
- 支払い一覧サマリーの対象月は既存の日付・月ルールに従う。
- カテゴリ別予算は月次予算や全体予算の仕様見直しへ広げない。
- 支払い登録・更新フローに予算入力を追加しない。
- DB/API/RLS/RPC/migration が必要な場合は、Design / Plan で影響範囲、操作単位、検証方針を整理する。

## 受け入れ条件

- AC-1: カテゴリ登録時に、カテゴリ別予算を一緒に登録できる。
- AC-2: カテゴリ登録時に、予算未入力でもカテゴリを登録できる。
- AC-3: カテゴリ更新時に、カテゴリ別予算を登録、更新、削除できる。
- AC-4: 予算未入力、予算削除、0円予算が混同されない。
- AC-5: カテゴリ削除時に、紐づくカテゴリ別予算が通常の有効表示から外れる。
- AC-6: カテゴリ一覧で、カテゴリ別予算の有無を確認できる。
- AC-7: カテゴリ一覧で、予算がある場合は金額を確認できる。
- AC-8: 支払い一覧サマリーで、カテゴリ別の対象月合計額を確認できる。
- AC-9: 支払い一覧サマリーで、予算があるカテゴリは予算との差分を確認できる。
- AC-10: 差分は、予算なし、予算削除、0円予算、計算不能、取得失敗を0差として扱わない。
- AC-11: スマホ幅では、支払い一覧サマリーのカテゴリ表示が1列で読める。
- AC-12: カテゴリと予算を同じ操作として扱う保存や削除で、片成功が完了済みに見えない。
- AC-13: 月次予算、全体予算、カテゴリ以外の予算、支払い登録・更新フローの仕様は変わらない。

## Q&Aログ

- Q: これまで作成した Requirements、Design、実装コードを参考にするか？
  - A: しない。この Requirements は Issue #1440、監督入力、選択 docs、現在有効な実装から作り直す。
- Q: 過去のカテゴリ別予算 migration や関数は、この Requirements の実装方針として扱うか？
  - A: 扱わない。履歴として存在する可能性は Design で調査してよいが、Requirements では現在ユーザーが必要とする体験と制約を定義する。
- Q: カテゴリ別予算は月別履歴を持つ前提か？
  - A: Issue だけでは確定しない。対象月サマリーで比較するため、基準月、履歴、過去/現在/未来への影響は Design で決める。
- Q: カテゴリ一覧やサマリーの具体的な layout を Requirements で決めるか？
  - A: 決めない。ユーザーが判断したい値と状態を定義し、列構成、component、copy は Design で決める。

## 技術的考慮事項

- 現行の有効 Web UI では、カテゴリ登録/更新フォームはカテゴリ名とピン留めを扱い、予算入力を扱わない。
- 現行のカテゴリ一覧はカテゴリ名、ピン留め、行操作を表示し、カテゴリ別予算を表示しない。
- 現行の支払い一覧サマリーはカテゴリ別支払い合計を表示し、カテゴリ別予算や差分を表示しない。
- 現行の支払い一覧サマリーは複数列表示を持つため、スマホ幅での1列読み順を Design で整理する必要がある。
- API migration には過去のカテゴリ別予算関連履歴があるため、Design / Plan では現在の最終 schema、RLS、RPC、削除済み履歴の扱いを確認する必要がある。
- カテゴリ登録/更新/削除と予算変更を同じ操作として見せる場合、Design / Plan で transaction boundary を決める必要がある。
