# Requirements / PRD: カテゴリなし支払いの文脈別表示を整理する

## 背景と課題

支払いカテゴリは任意であり、カテゴリ未設定の支払いはドメイン上「カテゴリなし」または「未分類」として扱う。カテゴリ削除時も関連する支払いはカテゴリ未設定になるため、カテゴリなし支払いは不明データ、取得失敗、削除済みカテゴリではなく、通常の支払い状態として扱う必要がある。

Issue #1480 では、カテゴリなし支払いを UI 上で文脈ごとに正しく読めるようにすることが求められている。現行英語 UI では、支払い一覧のカテゴリ絞り込みと月次サマリーなどの集計 bucket では `Uncategorized` を使い、登録・編集フォームと支払い詳細では未選択/カテゴリなし状態として `None` を使う。支払い一覧の行表示ではカテゴリ名が存在しないためカテゴリ表示を出さない。`Unknown` はカテゴリなしを表す文言として使わない。

PR #1490 のレビューでは、実在カテゴリ名として `None` や `Uncategorized` が使える場合、同じ文字列のシステムラベルと区別できないことが指摘された。また、`No category` のような補助テキストで区別するのではなく、system label 側の視覚表現で実在カテゴリ名と区別する方針が示された。更新済み Domain UI policy では、ユーザー入力のエンティティ名を予約語として禁止せず、同じ文字列の system label を追加説明テキストや非表示ラベルで区別せず、同じ表示単位内の色、文字トーン、表示階層、状態用 variant などの視覚表現で状態・bucket・fallback だと分かるようにすることが求められる。

## Current State / Current Gaps

- 支払いカテゴリは任意である。
- カテゴリ未設定の支払いはカテゴリなしとして扱う。
- カテゴリ未設定の支払いは未分類として扱う。
- 月次カテゴリ別支出合計では、カテゴリ未設定の支払いは未分類の月次支出合計として扱う。
- Web UI では、カテゴリなし、未選択、不明、削除済み、取得失敗を混同しない必要がある。
- Issue #1480 は、カテゴリなし支払いの表示文言を文脈ごとに分け、カテゴリなしを `Unknown` と表示しないことを求めている。
- カテゴリ名はユーザー入力のエンティティ名であり、`None` や `Uncategorized` をカテゴリ名として使える。これらの文字列がカテゴリなし状態や集計 bucket の system label と一致しても、予約語として禁止せず、追加説明テキストや非表示ラベルではなく視覚表現で区別できる必要がある。

## 対象ユーザーと利用シーン

- 対象ユーザー: savings の Web UI で支払い登録、編集、詳細確認、一覧確認、カテゴリ絞り込み、月次サマリー確認を行う認証済みユーザー。
- 登録・編集時: カテゴリを選ばない状態を、カテゴリ名ではなく未選択状態として理解したい。
- 支払い詳細確認時: カテゴリなし状態を、不明データではなくカテゴリ未設定の状態として確認したい。
- 支払い詳細確認時: 実在カテゴリ名 `None` とカテゴリなし状態の `None` を、追加説明テキストではなく視覚表現で区別して確認したい。
- 支払い一覧確認時: カテゴリなし支払いを、存在しないカテゴリ名や状態ラベルに惑わされず確認したい。
- 支払い一覧のカテゴリ絞り込み時: カテゴリなし支払いだけを絞り込む条件を識別したい。
- 月次サマリー確認時: 対象月のカテゴリなし支払い合計を、未分類の集計 bucket として確認したい。
- 月次サマリー確認時: 実在カテゴリ名 `Uncategorized` の集計行とカテゴリなし集計 bucket の `Uncategorized` を、追加説明テキストではなく視覚表現で区別して確認したい。

## ユーザーストーリー

- 認証済みユーザーとして、支払い登録・編集フォームでカテゴリを選ばないとき、未選択状態を `None` として理解したい。そうすることで、カテゴリ名を入力していない状態だと分かる。
- 認証済みユーザーとして、カテゴリなし支払いの詳細を見るとき、カテゴリ欄を `None` として読みたい。そうすることで、不明データや取得失敗ではなくカテゴリなし状態だと理解できる。
- 認証済みユーザーとして、支払い一覧でカテゴリなし支払いを見るとき、カテゴリ表示を見たくない。そうすることで、存在しないカテゴリ名や状態ラベルと誤解しない。
- 認証済みユーザーとして、支払い一覧のカテゴリ絞り込みで `Uncategorized` を選び、カテゴリなし支払いを探したい。
- 認証済みユーザーとして、月次サマリーでカテゴリなし支払い合計を `Uncategorized` として確認したい。
- 認証済みユーザーとして、カテゴリが設定された支払いはこれまでどおりカテゴリ名と絞り込み、集計に反映されてほしい。
- 認証済みユーザーとして、実在カテゴリ名とカテゴリなし状態または bucket が同じ文字列になっても、補助文言に頼らず、それぞれが何を表すかを区別したい。

## 機能要件

### FR-1: 支払い一覧のカテゴリ絞り込みではカテゴリなし条件を `Uncategorized` とする

- 支払い一覧のカテゴリ絞り込みでは、カテゴリなし支払いを選ぶ条件名を `Uncategorized` とする。
- `Uncategorized` はカテゴリなし支払いを検索・絞り込みする bucket 名であり、ユーザー作成カテゴリではない。
- `Unknown` はカテゴリなし支払いの絞り込み条件名として使わない。

### FR-2: 登録・編集フォームではカテゴリ未選択状態を `None` とする

- 登録・編集フォームでは、カテゴリを選ばない状態を `None` として扱う。
- `None` はカテゴリ未選択状態であり、カテゴリ名ではない。
- 登録・編集フォームの `None` は、カテゴリなしを `Unknown` として扱うことを意味しない。

### FR-3: 支払い詳細ではカテゴリなし状態を `None` として表示する

- 支払い詳細では、カテゴリなし状態を `None` として表示する。
- 支払い詳細の `None` はカテゴリなし状態であり、ユーザー作成カテゴリ名ではない。
- 支払い詳細では、カテゴリなし状態の表示名として `Unknown` を使わない。
- 実在カテゴリ名が `None` の支払いと、カテゴリなし状態の支払いは、支払い詳細で同じ意味に見えてはいけない。
- カテゴリなし状態の `None` を実在カテゴリ名 `None` と区別するために、`No category` のような追加説明テキストや ARIA label などの非表示ラベルへの追加説明語を使わない。

### FR-4: 支払い一覧の行表示ではカテゴリなし支払いにカテゴリ表示を出さない

- 支払い一覧の行表示では、カテゴリなし支払いにカテゴリ名や状態ラベルを表示しない。
- 支払い一覧の行表示では、カテゴリなし支払いに `None`、`Uncategorized`、`Unknown` を表示しない。
- カテゴリなし支払いでも、支払い自体の既存情報は確認できる。

### FR-5: 月次サマリーなどの集計ではカテゴリなし支払いを `Uncategorized` に含める

- 月次サマリーなどの集計では、カテゴリなし支払いを `Uncategorized` の集計 bucket に含める。
- 複数のカテゴリなし支払いは、対象月内で `Uncategorized` に合算する。
- 対象月、金額計算、カテゴリ紐付き支払いの集計は変更しない。
- 実在カテゴリ名が `Uncategorized` の集計と、カテゴリなし支払いの集計 bucket は、月次サマリーなどの集計で同じ意味に見えてはいけない。
- カテゴリなし集計 bucket の `Uncategorized` を実在カテゴリ名 `Uncategorized` と区別するために、`No category` のような追加説明テキストや ARIA label などの非表示ラベルへの追加説明語を使わない。

### FR-6: 既存カテゴリ紐付き支払いの表示、絞り込み、集計を維持する

- カテゴリ紐付き支払いは、既存どおりカテゴリ名を確認できる。
- カテゴリ紐付き支払いは、支払い一覧のカテゴリ絞り込みで既存どおり該当カテゴリ条件に含まれる。
- カテゴリ紐付き支払いは、月次サマリーで既存どおり該当カテゴリに集計される。
- 実在カテゴリ名が `Unknown`、`None`、`Uncategorized` の場合でも、そのカテゴリ名とカテゴリなし状態を混同しない。

### FR-7: ユーザー入力名とシステムラベルが衝突しても予約語にしない

- `None`、`Uncategorized`、`Unknown` はカテゴリ名として予約しない。
- ユーザー入力のカテゴリ名と、カテゴリなし状態、集計 bucket、fallback ラベルが同じ文字列になっても、カテゴリ名を入力不可にして解決しない。
- 同じ文字列のシステムラベルを表示する場合は、それが実在カテゴリ名ではなく状態、bucket、fallback のどれを表すかをユーザーが区別できる必要がある。
- 同じ文字列のシステムラベルは、追加説明テキスト、補助ラベル、ARIA label などの非表示ラベルへの追加説明語ではなく、同じ表示単位内の視覚表現で区別する。

## 非機能要件と制約

- ドメイン概念は「カテゴリなし」または「未分類」であり、`Uncategorized` や `None` を固定ドメイン概念として扱わない。
- `Uncategorized` は絞り込み/集計 bucket の現行英語 UI 表示として扱う。
- `None` は登録・編集/詳細のカテゴリなし状態の現行英語 UI 表示として扱う。
- `None` はカテゴリ名として予約しない。実在カテゴリ名 `None` とカテゴリなし状態の `None` を同じ意味に見せない。
- `Uncategorized` はカテゴリ名として予約しない。実在カテゴリ名 `Uncategorized` とカテゴリなし集計 bucket の `Uncategorized` を同じ意味に見せない。
- `Unknown` をカテゴリなし表示名として使わない。
- 不明参照、取得不能、削除済みカテゴリなどの fallback 文脈は、カテゴリなし状態と混同しない。
- 同じ文字列の system label は、実在カテゴリ名と同じ意味に見えてはいけない。
- system label と実在カテゴリ名の区別に、`No category` のような追加説明テキスト、補助ラベル、ARIA label などの非表示ラベルへの追加説明語を使わない。
- system label と実在カテゴリ名の区別は、同じ表示単位内の視覚表現で成立させる。Requirements では具体的な色、variant、component、配置は固定しない。
- Issue #1480 と選択 refs から追跡できない成功条件を追加しない。

## Q&Aログ

- Q: `Uncategorized` はドメイン概念か？
  - A: 違う。ドメイン概念は「カテゴリなし」または「未分類」であり、`Uncategorized` は支払い一覧のカテゴリ絞り込みと集計 bucket の現行英語 UI 表示である。
- Q: `None` はカテゴリ名か？
  - A: 違う。登録・編集フォームと支払い詳細でカテゴリ未選択/カテゴリなし状態を示す現行英語 UI 表示である。
- Q: 支払い一覧の行表示でも `None` を出すか？
  - A: 出さない。一覧行はカテゴリ名表示の文脈であり、カテゴリなし支払いにはカテゴリ名が存在しないため表示しない。
- Q: 月次サマリーでは `None` と `Uncategorized` のどちらを使うか？
  - A: `Uncategorized` を使う。集計 bucket を識別する文脈だから。
- Q: `Unknown` をカテゴリなし表示に使えるか？
  - A: 使わない。`Unknown` は不明参照、取得不能、削除済みカテゴリなどの別 fallback 文脈に限る。
- Q: i18n 基盤を導入するか？
  - A: しない。今回扱うのは現行英語 UI 表示の要求整理であり、翻訳管理方式は対象外。
- Q: `None` や `Uncategorized` をカテゴリ名として禁止するか？
  - A: しない。これらは UI ラベルと衝突し得るが、ユーザー入力のカテゴリ名を予約語として禁止せず、UI 上で実在カテゴリ名とシステムラベルを区別できるようにする。
- Q: 実在カテゴリ名とシステムラベルが同じ文字列になる場合、Requirements で具体的な UI 表現まで決めるか？
  - A: 決めない。Requirements では、ユーザーが実在カテゴリ名、カテゴリなし状態、集計 bucket、fallback の意味を区別できることを成功条件にする。具体的な色、variant、component、配置は Design / Plan で決める。ただし、`No category` のような追加説明テキストや非表示ラベルへの追加説明語は使わない。
- Q: `No category` のような補助テキストで区別するか？
  - A: しない。可視テキストだけでなく、ARIA label などの非表示ラベルにも追加説明語を入れない。system label 自体の視覚表現で区別する。
- Q: `color: gray` を Requirements で固定するか？
  - A: 固定しない。レビューコメント上の例として扱い、Requirements では視覚表現で区別する必要だけを定義する。

## 技術的考慮事項

- 支払いカテゴリは任意であり、カテゴリ未設定の支払いはカテゴリなしとして扱う。
- カテゴリ削除時、関連する支払いはカテゴリ未設定になる。
- 月次カテゴリ別支出合計では、カテゴリ未設定支払いは未分類の月次支出合計として扱う。
- カテゴリ名はユーザー入力のエンティティ名であり、`None`、`Uncategorized`、`Unknown` を予約語として扱う要件はない。
- Domain UI policy は、同じ文字列の system label を実在エンティティ名と同じ意味に見せず、追加説明テキストや非表示ラベルではなく同じ表示単位内の視覚表現で区別することを求めている。
- Requirements では実装層、コンポーネント、具体的なテストコード、状態管理方式を決めない。
- 後続の Design / Plan では、文脈ごとの表示文言がどの状態、bucket、fallback、実在カテゴリ名に対応するかを整理する。
- 後続の Build / Verify で Web アプリケーションコードを変更する場合は、AGENTS.md の Web verification batch が必要になる。

## Stop 条件

- 要求を成立させるために前回実装コード、前回 UI 挙動、現在 diff 形状、前回 Design 判断を根拠にする必要がある。
- Issue #1480、PR #1490 レビュー指摘、更新済み Domain UI policy、または選択 refs から追跡できない表示状態、copy、UI構造、i18n基盤、DB/API変更が必要になる。
- `Uncategorized` を固定ドメイン概念またはカテゴリ entity として扱う必要がある。
- `None`、`Uncategorized`、`Unknown` をカテゴリ名の予約語として禁止する必要がある。
- 実在カテゴリ名と system label の区別に `No category` のような追加説明テキスト、補助ラベル、ARIA label などの非表示ラベルへの追加説明語が必要になる。
- 実在カテゴリ名と system label の区別を Requirements で具体的な色、配置、component、badge、grouping まで決める必要がある。
- カテゴリなし、不明、削除済み、取得失敗、実在カテゴリ名を要件内で分けられない。
- Requirements / PRD が選択したルール・ポリシーに違反する可能性がある。
