# Requirements / PRD: 支払い一覧とサマリーの未カテゴリ表示を調整する

## 背景と課題

支払いカテゴリは任意であり、カテゴリ未設定の支払いはドメイン上の通常ケースとして扱う。カテゴリ削除後の関連支払いもカテゴリ未設定になるため、未カテゴリ支払いは異常値や取得失敗ではなく、ユーザーが確認する支払い明細と月次集計に含まれる。

Issue #1480 では、カテゴリの紐付きがない支払い情報について、支払い一覧でカテゴリ表示を出さないことと、サマリー集計では `Unknown` ではなく `Uncategorized` として集計することが求められている。この Requirements / PRD は Issue #1480 を最終仕様ではなく、意図、境界、成功条件の入力として扱う。

## Current State / Current Gaps

- 支払いはカテゴリなしで存在できる。
- カテゴリ未設定の支払いは、支払い明細としては通常の支払いであり、金額と日付を持つ。
- 月次カテゴリ別支出合計では、カテゴリ未設定の支払いも集計対象に含まれる。
- Issue #1480 は、支払い一覧ではカテゴリ未紐付き支払いにカテゴリ表示を出さないこと、サマリーではカテゴリ未紐付き支払いを `Unknown` ではなく `Uncategorized` として集計することを調整対象としている。
- `Unknown`、未設定、参照先不明、取得失敗が同じ意味に見えると、ユーザーは支払い自体の状態や集計の信頼性を誤解しやすい。

## 対象ユーザーと利用シーン

- 対象ユーザー: savings の Web UI で支払い一覧と月次サマリーを確認する認証済みユーザー。
- 支払い一覧確認時: 個々の支払いについて、カテゴリが設定されているかどうかを余計なカテゴリ名や状態ラベルに惑わされず確認したい。
- サマリー確認時: 対象月のカテゴリ別支出合計の中で、カテゴリ未設定の支払い合計を他のカテゴリ別合計と同じ集計単位として把握したい。
- カテゴリ削除後の確認時: 関連支払いがカテゴリ未設定になった場合でも、支払いが消えたようにも、取得失敗のようにも見えない状態で確認したい。

## ユーザーストーリー

- 認証済みユーザーとして、カテゴリ未設定の支払いを一覧で見るとき、存在しないカテゴリ名や状態ラベルをカテゴリ表示として見たくない。そうすることで、その支払いにカテゴリが紐付いていないことを自然に理解できる。
- 認証済みユーザーとして、月次サマリーでカテゴリ未設定の支払い合計を `Uncategorized` として確認したい。そうすることで、カテゴリが未設定の支出合計を不明データや取得失敗ではなく、集計された支出として読める。
- 認証済みユーザーとして、カテゴリが設定された既存支払いはこれまでどおりカテゴリ名と集計に反映されてほしい。そうすることで、今回の調整によって既存のカテゴリ別確認が壊れていないと判断できる。

## スコープ

### 対象

- 支払い一覧におけるカテゴリ未紐付き支払いのカテゴリ表示。
- 月次サマリーにおけるカテゴリ未紐付き支払いのカテゴリ別集計名。
- カテゴリ紐付き支払いの一覧表示とサマリー集計を維持すること。
- `Unknown`、`Uncategorized`、未設定、不明、取得失敗の意味を混同しない要求整理。

### 対象外

- カテゴリ作成、更新、削除の仕様変更。
- 支払い作成、更新、削除の仕様変更。
- カテゴリ未設定支払いを自動分類すること。
- 新しいカテゴリ、状態ラベル、フィルター、並び順、集計単位の追加。
- 具体的な UI 配置、コンポーネント、色、variant、最終的なレイアウトの決定。
- DB、API、認証、権限の変更。
- この Goal 内での Design Doc 作成、実装、PR 作成。

## 機能要件

### FR-1: 支払い一覧ではカテゴリ未紐付き支払いにカテゴリ表示を出さない

- カテゴリが紐付いていない支払い情報では、支払い一覧にカテゴリ表示を出さない。
- カテゴリ未紐付き支払いに対して、`Unknown`、`Uncategorized`、その他の状態ラベルをカテゴリ表示として出さない。
- カテゴリ未紐付き支払いでも、支払い自体の金額、日付、メモなど、既存の支払い情報は引き続き確認できる。
- この Requirements では、カテゴリ表示がない場合の余白、配置、アクセシブルな名前、具体的な表示構造は固定しない。Design Doc で既存 UI の文脈に合わせて決める。

### FR-2: サマリーではカテゴリ未紐付き支払いを `Uncategorized` として集計する

- 月次サマリーのカテゴリ別集計では、カテゴリが紐付いていない支払い情報を `Uncategorized` の集計単位に含める。
- `Uncategorized` は、カテゴリ未設定の支払い合計を表す集計名として扱う。
- `Uncategorized` は、新しいユーザー作成カテゴリ、削除済みカテゴリ、取得失敗、不明参照を表す状態ラベルとして扱わない。
- カテゴリ未紐付き支払いが複数ある場合、対象月内の該当支払い金額は `Uncategorized` として合計される。

### FR-3: 既存のカテゴリ紐付き支払いの表示と集計を維持する

- カテゴリが紐付いた支払いは、支払い一覧で既存どおりカテゴリを確認できる。
- カテゴリが紐付いた支払いは、月次サマリーで既存どおり該当カテゴリの集計に含まれる。
- 今回の調整によって、既存カテゴリ名、カテゴリ別合計額、対象月の集計範囲を変更しない。
- 既存のカテゴリ紐付き支払いとカテゴリ未紐付き支払いが混在しても、それぞれの表示と集計の意味が区別できる。

### FR-4: 未設定、不明、取得失敗を混同しない

- カテゴリ未設定は、カテゴリが紐付いていない支払いの通常状態として扱う。
- `Unknown` は、Issue #1480 の対象であるサマリー集計名としては使わない。
- 参照先不明、削除済みカテゴリ、取得失敗を今回の Requirements で新しい表示状態として追加しない。
- Design / Plan で不明参照や取得失敗の扱いが必要だと分かった場合は、今回の要求に混ぜず、別判断として止める。

## 非機能要件と制約

- Issue #1480 の「カテゴリ表示を出さない」「`Unknown` ではなく `Uncategorized`」という表現を弱めない。
- Issue の例を固定 UI レイアウトやコンポーネント選定として扱わない。
- ユーザーが変更できるカテゴリと、システムが付与する集計名を混同させない。
- 支払い一覧では、カテゴリ未設定を目立つ状態表示として追加しない。
- サマリーでは、カテゴリ未設定の合計が集計対象から落ちたように見せない。
- 新規依存、DB/API/RLS/RPC/migration の変更はこの Requirements の前提にしない。

## Q&Aログ

- Q: Issue #1480 は最終仕様として扱うか？
  - A: 扱わない。Issue は Requirements / PRD の入力として扱い、意図、境界、成功条件を検証可能な要求へ変換する。
- Q: 支払い一覧でカテゴリ未設定を `Uncategorized` と表示するか？
  - A: しない。Issue #1480 は一覧ではカテゴリ表示を出さないことを求めている。`Uncategorized` はサマリー集計名として扱う。
- Q: `Unknown` を別の意味で残すか？
  - A: 今回のサマリー集計名としては残さない。不明参照や取得失敗の扱いが別途必要な場合は、今回の要求に混ぜず Design / Plan で停止判断にする。
- Q: `Uncategorized` はユーザーが作成したカテゴリか？
  - A: 違う。カテゴリ未設定支払いの月次集計単位を識別するための集計名であり、新しいカテゴリ作成を意味しない。
- Q: カテゴリ削除や支払い編集の仕様も変えるか？
  - A: 変えない。カテゴリ未設定になる既存ドメイン前提を維持し、その表示と集計名だけを対象にする。
- Q: 具体的な UI 配置やコンポーネントを Requirements で決めるか？
  - A: 決めない。Design Doc で既存 UI、アクセシビリティ、表示階層に合わせて決める。

## 技術的考慮事項

- 支払いカテゴリは任意であり、カテゴリ未設定の支払いはカテゴリなしとして扱う。
- 月次カテゴリ別支出合計では、カテゴリ未設定の支払いは未分類の月次支出合計として扱う。
- カテゴリ削除時、関連する支払いはカテゴリ未設定になる。
- Requirements 上の `Uncategorized` は、ドメイン文書上の「未分類」に対応する Web UI の集計名として扱う。
- Design / Plan Goal では、既存の支払い一覧と月次サマリーにおいて、カテゴリ未設定、カテゴリ紐付き、集計名、取得失敗の区別を確認する。
- Build / Verify Goal では、Web のアプリケーションコード変更が想定されるため、AGENTS.md の Web verification batch が必要になる。
