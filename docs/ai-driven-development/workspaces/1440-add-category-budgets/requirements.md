# Requirements / PRD: カテゴリ別予算機能を追加する

## 背景と課題

設定画面ではカテゴリの作成、名前変更、削除、ピン留め状態の設定ができる。しかし、カテゴリごとの予算は現在のカテゴリ設定体験では扱えない。

支払い一覧では、対象月の月次合計とカテゴリ別支出額を確認できる。カテゴリ別サマリーは支出状況の把握に役立つが、カテゴリごとの予算が表示されないため、ユーザーは「このカテゴリの支出が予算内か」を同じ画面で判断できない。

スマホでは、支払い一覧サマリのカテゴリ表示が現在 2 列で表示される。カテゴリ名、支出額、予算情報を同時に確認する場合、2 列表示のままでは読みづらくなりやすい。

この要求は GitHub Issue #1440 を入力として整理する。Issue は最終仕様ではなく、Requirements / PRD のための意図、境界、成功条件として扱う。

## 対象ユーザーと利用シーン

- 対象ユーザー: 個人の支払い・貯蓄管理をこのアプリで行う認証済みユーザー。
- 利用シーン:
  - 設定画面でカテゴリを作成するとき、そのカテゴリの予算も一緒に設定する。
  - 設定画面のカテゴリ一覧で、各カテゴリにカテゴリ別予算が設定されているかを確認する。
  - 設定画面でカテゴリを見直すとき、カテゴリ名やピン留め状態と同じ流れでカテゴリ別予算を登録、更新、削除する。
  - 支払い一覧で対象月のカテゴリ別支出額を確認しながら、カテゴリ別予算も確認する。
  - スマホで支払い一覧サマリを見たとき、カテゴリごとの支出額と予算情報を読み取る。
- 関連する現行体験:
  - 設定画面のカテゴリ作成フォームはカテゴリ名とピン留め状態を扱う。
  - 設定画面のカテゴリ更新フォームはカテゴリ名とピン留め状態を扱う。
  - カテゴリ作成とピン留め、カテゴリ更新とピン留め変更は `create_category_with_pin` / `update_category_with_pin` RPC で 1 操作として扱われている。
  - 設定画面からカテゴリ削除ができる。
  - 支払い一覧のカテゴリ別サマリーは、カテゴリ支出額と未分類支出額を表示する。
  - 支払い一覧のカテゴリ別サマリーはピン留めカテゴリを優先し、初期表示を最大 3 件に抑え、`Show more` で残りを表示する。
  - 現在の有効 DB 型には `category_budgets` は含まれていない。古い migration にはカテゴリ別予算テーブルと関連関数が存在するが、`20260528000000_remove_category_budgets.sql` で削除済みである。

## ユーザーストーリー

- 認証済みユーザーとして、カテゴリを作成するときにカテゴリ別予算も一緒に登録したい。そうすることで、カテゴリを追加した直後からそのカテゴリの支出と予算を見比べられる。
- 認証済みユーザーとして、設定画面のカテゴリ一覧で各カテゴリの予算状態を確認したい。そうすることで、カテゴリ更新操作に入る前に、どのカテゴリに予算があるかを把握できる。
- 認証済みユーザーとして、カテゴリを編集するときにカテゴリ別予算も登録、更新、削除したい。そうすることで、カテゴリ設定を見直す流れの中で予算も保守できる。
- 認証済みユーザーとして、カテゴリを削除したときに紐づくカテゴリ別予算も残らないでほしい。そうすることで、削除済みカテゴリの予算が支払い一覧や内部状態に残らない。
- 認証済みユーザーとして、支払い一覧のカテゴリ別サマリーでカテゴリ別予算を確認したい。そうすることで、カテゴリごとの支出状況を予算と合わせて判断できる。
- 認証済みユーザーとして、スマホではカテゴリ別サマリーを 1 列で読みたい。そうすることで、カテゴリ名、支出額、予算情報が詰まりすぎず確認できる。

## スコープ

### 対象

- 設定画面のカテゴリ作成時に、カテゴリ別予算を一緒に登録できること。
- 設定画面のカテゴリ一覧で、カテゴリ別予算状態を確認できること。
- 設定画面のカテゴリ更新時に、カテゴリ別予算を登録、更新、削除できること。
- カテゴリ削除時に、そのカテゴリに紐づくカテゴリ別予算が有効な予算として残らないこと。
- 支払い一覧のカテゴリ別サマリーで、対象月のカテゴリ別支出額とカテゴリ別予算を確認できること。
- スマホの支払い一覧サマリでは、カテゴリ表示を 1 列にすること。
- カテゴリ別予算の保存失敗、更新失敗、削除失敗をユーザーが認識できること。
- カテゴリ本体、ピン状態、カテゴリ別予算状態を永続データ上の別関心として扱いながら、ユーザーから 1 回の保存に見える操作では片方だけ成功した状態を避けること。

### 対象外

- 月次予算や全体予算の仕様見直し。
- カテゴリ以外を対象にした予算機能の追加。
- 支払い登録・更新フローへの予算入力の追加。
- カテゴリ別予算の履歴一覧や監査ログ機能。
- カテゴリ別予算の任意並び替え。
- Requirements / PRD 作成 Goal 内での Design Doc 作成、実装、PR 作成。

## 機能要件

### FR-1: カテゴリ作成時にカテゴリ別予算を一緒に登録できる

- ユーザーは設定画面のカテゴリ作成フォームで、カテゴリ名、ピン留め状態に加えてカテゴリ別予算を入力できる。
- カテゴリ別予算は任意入力として扱える。カテゴリだけを作成し、カテゴリ別予算を設定しない操作もできる。
- カテゴリ別予算を入力してカテゴリ作成した場合、作成されたカテゴリに紐づく予算として保存される。
- カテゴリ作成とカテゴリ別予算登録は、ユーザーからは 1 回の作成操作として見える。
- カテゴリ作成だけ成功し、カテゴリ別予算登録だけ失敗した状態を成功扱いにしない。
- カテゴリ別予算登録に失敗した場合、ユーザーはカテゴリ作成が完了していない、または予算を含む保存が失敗したことを認識できる。

### FR-2: 設定画面でカテゴリ別予算状態を確認し、登録、更新、削除できる

- ユーザーは設定画面のカテゴリ一覧で、各カテゴリのカテゴリ別予算状態を確認できる。
- カテゴリ別予算が設定されているカテゴリでは、カテゴリ一覧でユーザーがそのカテゴリの予算額を確認できる。
- カテゴリ別予算が未設定、または予算なし状態のカテゴリでは、カテゴリ一覧でユーザーが金額あり予算として誤認しない。
- 0 円予算が設定されているカテゴリでは、カテゴリ一覧で 0 円予算を予算なしと同一視しない。
- ユーザーは設定画面のカテゴリ更新フォームで、現在のカテゴリ別予算状態を確認できる。
- カテゴリ別予算が未設定のカテゴリに対して、カテゴリ更新時に予算を登録できる。
- カテゴリ別予算が設定済みのカテゴリに対して、カテゴリ更新時に予算額を更新できる。
- カテゴリ別予算が設定済みのカテゴリに対して、カテゴリ更新時に予算を削除できる。
- カテゴリ名、ピン留め状態、カテゴリ別予算を同じ保存操作で変更する場合、片方だけが成功した状態を成功扱いにしない。
- カテゴリ別予算の登録、更新、削除に失敗した場合、ユーザーは失敗を認識でき、画面上で成功後の状態として扱われない。

### FR-3: カテゴリ削除時に紐づくカテゴリ別予算も残らない

- カテゴリ削除が成功した場合、そのカテゴリに紐づくカテゴリ別予算は有効なカテゴリ別予算として参照されない。
- 削除済みカテゴリのカテゴリ別予算は、支払い一覧のカテゴリ別サマリーに表示されない。
- 削除済みカテゴリのカテゴリ別予算は、設定画面のカテゴリ一覧やカテゴリ更新フォームで参照されない。
- カテゴリ削除によって、支払い自体は削除されない。
- カテゴリ削除とカテゴリ別予算削除のどちらか片方だけが成功した状態を成功扱いにしない。
- カテゴリ削除に失敗した場合、紐づくカテゴリ別予算だけが削除済みとして扱われない。

### FR-4: 支払い一覧のカテゴリ別サマリーでカテゴリ別予算を確認できる

- 支払い一覧のカテゴリ別サマリーでは、対象月のカテゴリ別支出額に加えて、カテゴリ別予算が設定されているかを確認できる。
- カテゴリ別予算が設定されているカテゴリでは、ユーザーがそのカテゴリの予算額を確認できる。
- カテゴリ別予算が未設定、または予算なし状態のカテゴリでは、ユーザーが予算がない状態を誤って金額あり予算として認識しない。
- カテゴリ別予算の表示は、現在のカテゴリ別サマリーのピン留め優先、最大 3 件の初期表示、`Show more` による追加表示と矛盾しない。
- 未分類 `Unknown` 行はカテゴリに紐づく予算を持たない。未分類行に予算を表示するかどうかではなく、カテゴリ別予算の対象外として扱う。
- 支払い一覧のサマリー取得またはカテゴリ別予算取得に失敗した場合、ユーザーは失敗を認識でき、古い予算状態を成功後の状態として扱い続けない。

### FR-5: スマホでは支払い一覧サマリのカテゴリ表示を 1 列にする

- スマホ幅では、支払い一覧サマリのカテゴリ表示を 1 列で表示する。
- スマホ幅の 1 列表示では、カテゴリ名、カテゴリ別支出額、カテゴリ別予算情報が読み取れる。
- スマホ幅で、カテゴリ行のテキストや金額が重なったり、ボタンや次の行を不自然に押しつぶしたりしない。
- デスクトップ幅の表示列数は、Design Doc で既存の 2 列表示とカテゴリ別予算追加後の読みやすさを比較して決める。

### FR-6: カテゴリ別予算の期間、履歴、削除後状態を扱える

- 支払い一覧のカテゴリ別サマリーは対象月を持つため、カテゴリ別予算も対象月に対して表示できる必要がある。
- カテゴリ別予算を月別、期間別、または現在値として保存するかは、この Requirements / PRD では固定しない。
- カテゴリ別予算に有効開始日や履歴を持たせる場合、`docs/policies/temporal-data.md` に沿って、現在有効な状態と過去時点で閲覧すべき状態を分けて扱う。
- カテゴリ別予算の削除が「予算なし」状態を意味する場合、過去のカテゴリ別予算が暗黙に復活しない必要がある。
- カテゴリ別予算の削除が単なる未設定への戻しで足りる場合でも、支払い一覧の対象月でユーザーが予算ありと誤認しない必要がある。
- 0 円予算を許可する場合、0 円予算と予算なしを区別できる必要がある。

## 代表シナリオ

2026/10/01 時点で、ユーザーが次のカテゴリを持っている場合:

- 食費: ピン留めあり、2026/10 の支出額 42,000 円
- 日用品: ピン留めなし、2026/10 の支出額 8,000 円
- 交通費: ピン留めなし、2026/10 の支出額 12,000 円

期待する挙動:

- ユーザーは設定画面で「食費」のカテゴリ別予算を登録できる。
- 「食費」に 50,000 円のカテゴリ別予算が設定されている場合、設定画面のカテゴリ一覧で食費の予算 50,000 円を確認できる。
- 「日用品」にカテゴリ別予算がない場合、設定画面のカテゴリ一覧で日用品を金額あり予算として誤認しない。
- 「食費」に 50,000 円のカテゴリ別予算が設定されている場合、2026/10 の支払い一覧サマリで食費の支出額 42,000 円と予算 50,000 円を確認できる。
- 「日用品」にカテゴリ別予算がない場合、2026/10 の支払い一覧サマリで日用品を金額あり予算として誤認しない。
- ユーザーが「食費」のカテゴリ別予算を削除した場合、2026/10 の支払い一覧サマリで食費の予算が金額ありとして表示されない。
- カテゴリ別予算を履歴として扱う設計を採用する場合、削除後に過去の食費予算が 2026/10 の有効予算として暗黙に復活しない。
- ユーザーが「交通費」カテゴリを削除した場合、交通費のカテゴリ別予算は支払い一覧サマリや設定画面に残らない。ただし、交通費に紐づいていた支払い自体は削除されない。
- スマホ幅では、カテゴリ別サマリーが 1 列で表示され、カテゴリ名、支出額、予算情報を読み取れる。

## 非機能要件と制約

- カテゴリ別予算は Book 所有境界と矛盾しない。
- カテゴリ別予算はカテゴリに紐づく予算として扱い、カテゴリ以外を対象にしない。
- 金額は既存の金額ドメインルールに従い、整数かつ 0 以上として扱う。
- 0 円予算を許可する場合、0 円予算は予算なしと同一視しない。
- カテゴリ本体、ピン状態、カテゴリ別予算状態は永続データ上の別関心として扱う。
- ユーザーから 1 回の保存に見えるカテゴリ作成・更新では、カテゴリ本体、ピン状態、カテゴリ別予算状態の片方だけが成功した状態を避ける。
- 設定画面のカテゴリ一覧とカテゴリ更新フォームで表示されるカテゴリ別予算状態は、同じ現在状態に基づいている必要がある。
- 支払い一覧のカテゴリ別サマリーで 1 つの表示として見える場合、カテゴリ支出額、ピン状態、カテゴリ別予算状態は同じ対象月に対して整合している必要がある。
- 既存の月次予算、全体予算、支払い作成、支払い更新の主要挙動を壊さない。
- 新規依存の追加はこの要求の前提にしない。
- DB 変更や API 仕様変更が必要な場合、Design Doc で影響範囲、移行方針、互換性、検証方針を明確にする。

## Q&Aログ

- Q: Issue #1440 は最終仕様として扱うか？
  - A: 扱わない。Issue は Requirements / PRD の入力として扱い、既存コード・既存ドキュメントから要求と受け入れ条件へ展開する。
- Q: 現在の有効 DB モデルに `category_budgets` は存在するか？
  - A: 存在しない。古い migration には `category_budgets` や `create_category_with_budget` / `update_category_with_budget` があるが、`20260528000000_remove_category_budgets.sql` で削除済みであり、現在の generated database types にも `category_budgets` は含まれていない。
- Q: カテゴリ別予算はカテゴリ本体の列として持つ要求にするか？
  - A: 固定しない。カテゴリ本体、ピン状態、カテゴリ別予算状態は永続データ上の別関心として扱う。ただし、UI から 1 回の保存に見える操作では片方だけ成功した状態を避ける要求にする。
- Q: カテゴリ別予算の期間や履歴はこの PRD で決めるか？
  - A: 最終的な保存方式は固定しない。ただし、支払い一覧サマリは対象月を持つため、カテゴリ別予算も対象月に対して表示できる必要がある。有効開始日や履歴を持つ設計を採用する場合は、過去状態の復活や過去表示の上書きを避ける必要がある。
- Q: カテゴリ別予算の削除は物理削除として固定するか？
  - A: 固定しない。削除後に対象月で金額あり予算として誤認されないことを要求とし、物理削除、予算なし状態、終端日、イベントなどの方式は Design Doc で比較する。
- Q: 0 円予算と予算なしは同じか？
  - A: 同じとは扱わない。0 円予算を許可する場合、0 円予算は金額あり予算、予算なしは予算差分を計算しない状態として区別できる必要がある。
- Q: 支払い一覧サマリでは予算額だけを表示するか、残額や超過額も表示するか？
  - A: この Requirements / PRD では、少なくともカテゴリ別予算を確認できることを要求する。残額や超過額を表示するかは、既存の月次予算表示と画面密度を踏まえて Design Doc で決める。
- Q: 未分類 `Unknown` にカテゴリ別予算を持たせるか？
  - A: 持たせない。未分類はカテゴリではないため、カテゴリ別予算の対象外として扱う。
- Q: スマホだけ 1 列にするか？
  - A: Issue の成功条件はスマホで 1 列にすること。デスクトップ幅の列数は、カテゴリ別予算追加後の読みやすさを Design Doc で確認する。
- Q: レビューコメント「カテゴリ一覧に予算が表示されていない」は Requirements / PRD で扱うか？
  - A: 扱う。カテゴリ別予算はこれから追加する機能であり、このコメントは追加後の設定画面カテゴリ一覧で予算状態を確認できるようにする表示要件として扱う。
- Q: カテゴリ一覧でカテゴリ別予算を設定できる要求にするか？
  - A: しない。カテゴリ一覧では予算状態を確認できることを要求し、登録、更新、削除はカテゴリ作成フォームまたはカテゴリ更新フォームの要求として扱う。
- Q: カテゴリ一覧で予算なしと未設定を区別して表示するか？
  - A: この Requirements / PRD では具体表示までは固定しない。ただし、どちらも金額あり予算として誤認されない必要があり、0 円予算は予算なしと区別できる必要がある。

## 技術的考慮事項

- 現行のカテゴリ作成は `create_category_with_pin` RPC を呼び、カテゴリ作成とピン留め作成を 1 操作として扱っている。
- 現行のカテゴリ更新は `update_category_with_pin` RPC を呼び、カテゴリ名更新とピン留め変更を 1 操作として扱っている。
- 現行のカテゴリ削除は `categories` table への delete として実装されている。
- 現行の `category_pins.category_id` は `categories(id) on delete cascade` であり、カテゴリ削除時にピン情報が残らない設計になっている。
- 過去の `category_budgets` は `categories(id) on delete restrict` だったが、現在は削除済みである。カテゴリ別予算を再導入する場合、カテゴリ削除時に予算が残らないための参照整合性を Design Doc で決める必要がある。
- 現行の `fetchCategoryTotals` は `categories` と `payments` を取得し、カテゴリごとの支出合計と未分類合計を返す。カテゴリ別予算は取得していない。
- 現行のカテゴリ別サマリーは `CategoryTotal` に `totalAmount`、`pinned`、`kind` を持つが、カテゴリ別予算状態は持たない。
- 現行のカテゴリ別サマリー表示は `chunkSize = 2` を既定にして 2 列に分割し、初期表示件数を 3 件にしている。
- 現行の月次予算は金額あり、予算なし、未設定を区別し、対象月末日以前に開始した最新状態で有効状態を決める。カテゴリ別予算に同じ考え方を適用するかは Design Doc で比較する。
- 現行の金額ドメインでは、金額は整数かつ 0 以上として扱う。
