# Web Design Rules

この文書は、Webアプリ全体に適用するドメイン非依存のデザインルールです。

画面固有の文言、業務概念、データ項目の順序、値のフォーマットはここでは扱いません。ドメインに依存するUI判断は [Domain UI Rules](./domain-ui-rules.md) に分離します。

AI agent は、UIを追加または変更するとき、この文書を既定値として扱います。個別のRequirements / Designで明示された差分がない限り、ここに書かれた密度、階層、余白、状態表現、レスポンシブ挙動から外れてはいけません。

## 適用順序

UI判断は、次の順序で決めます。

1. この文書のルール。
2. ドメイン固有ルール。
3. 既存の同種画面の実装。
4. UIライブラリの公式props、tokens、responsive API。
5. CSS Modulesによる局所調整。

上位のルールで表現できるものを、下位の方法で上書きしてはいけません。特に、直接style、固定px、独自色、独自の角丸、独自のshadow、独自のフォントサイズは、上位ルールでは表現できない局所的な制約がある場合だけ許可します。

## DataList Value Layout

- `DataList.Value` には、1つの主値または主値に直接付随する短い補助情報だけを置く。
- `DataList.Value` の中に、カテゴリ名、合計額、予算額のような複数の主要値を縦積みにしてはいけない。
- 複数レコード間で比較する主要値は、同じ `DataList.Value` 内にまとめず、行内の同じ列位置に分けて配置する。

## 基本原則

- UIは業務操作のための密度を優先する。装飾、ヒーロー表現、過度な余白、説明的なカードの追加をしてはいけない。
- 画面は「見る」「選ぶ」「入力する」「確認する」「保存する」の流れが最短で追える構成にする。
- 同じ意味のUIは同じ見た目にする。見た目が同じなら意味も同じにする。
- 状態表示と操作要素を混同させてはいけない。
- 値、ラベル、説明、エラー、操作、危険操作は視覚階層で必ず区別する。
- 余白は関係性を表す。近い要素ほど小さく、別グループほど大きくする。
- レイアウトはモバイルを1列の読み順として成立させ、PCで必要なときだけ列を増やす。
- 実装ではUIライブラリのpropsとtokensを優先し、CSSは補助に限定する。

## テーマとトークン

- アプリ全体のアクセント色、背景、角丸はテーマ設定に従う。
- 色を直接指定してはいけない。意味色はUIライブラリの `color` または既存tokenから選ぶ。
- 余白は `gap`、`padding`、`margin` 系props、または `--space-*` tokenで指定する。
- 角丸はテーマ既定に従う。個別の丸めは、丸形アイコンボタンなど形状そのものに意味がある場合だけ使う。
- shadowは浮上、overlay、temporary surfaceのためだけに使う。通常のページsectionを浮かせるために使ってはいけない。
- フォントファミリーを局所的に変えてはいけない。

## 文字階層

文字階層は次の意味に固定します。

| 用途             | 既定                                           | 使い方                                                    |
| ---------------- | ---------------------------------------------- | --------------------------------------------------------- |
| ページタイトル   | `size="6"` / heading                           | 画面内に1つ。ナビゲーション先または現在の作業領域を示す。 |
| セクション見出し | `size="4"` / `weight="medium"`                 | 一覧、設定グループ、フォームグループの先頭に置く。        |
| フィールドラベル | `size="2"` / `weight="bold"`                   | 入力、詳細、key-value表示の項目名に使う。                 |
| 標準本文         | `size="3"`                                     | 通常の説明、一覧内の標準値に使う。                        |
| 詳細値           | `size="4"`                                     | 詳細画面で、ラベルに対する現在値に使う。                  |
| 強調値           | `size="5"` または `size="6"` / `weight="bold"` | 一覧やサマリーで最も確認すべき値に使う。                  |
| 補助情報         | `size="2"` または `size="3"` / `color="gray"`  | 補足、空状態、placeholder相当の値に使う。                 |
| エラー           | `size="1"` または `size="2"` / `color="red"`   | フィールドエラー、ロード失敗、危険操作の警告に使う。      |

禁止事項:

- 見出しの見た目だけを作るために本文要素を大きくしてはいけない。
- 文字サイズをviewport幅に連動して連続的に変えてはいけない。
- 同じ画面内にページタイトル級の文字を複数置いてはいけない。
- 補助情報を本文色で表示してはいけない。
- エラーを色だけで表してはいけない。エラーテキストまたはalert相当の文言を必ず置く。

許可条件:

- responsive propsで段階的に文字サイズを変えてよいのは、モバイルで折り返しや押し出しが発生し、かつ意味階層が変わらない場合だけ。
- 強調値を `size="6"` より大きくしてよいのは、単一値を主役にする画面で、一覧密度を犠牲にしない場合だけ。

## 一覧パターン

一覧は、用途に応じて次のどれかに分類します。分類せずに新しい見た目を作ってはいけません。

| 用途               | 表現                            | レイアウト                                                       |
| ------------------ | ------------------------------- | ---------------------------------------------------------------- |
| 押せるレコード一覧 | 行全体を押せるカード状のsurface | 上段に補助情報と状態、中央に主テキスト、下段または右側に強調値。 |
| 設定・管理一覧     | 行区切りのgrid                  | PCはラベル列と操作列を分ける。モバイルは行内に操作を入れる。     |
| 履歴・数値比較     | table                           | ヘッダーを出し、数値は右寄せ。empty/error/loadingは行内で表示。  |
| key-valueサマリー  | key-value list                  | ラベル幅を揃え、値は短く、複数列化はPCまたは狭いサマリー内だけ。 |
| 短い状態の集合     | badge/list                      | 状態だけを表示する。操作を混ぜない。                             |

共通ルール:

- 一覧の1行は、主情報、補助情報、状態、操作を視覚的に分ける。
- 一覧内の主操作が行クリックである場合、行内に同じ遷移のボタンを追加してはいけない。
- 行内に複数操作がある場合、操作群は右端または行末にまとめる。
- 操作列は `flexShrink=0` 相当で潰れないようにする。
- ヘッダーとデータ行を持つgrid/table-likeな一覧では、同じ意味の列が同じ開始位置に揃うようにする。
- ヘッダーと行を別々のGridで実装する場合は、内容依存の `auto` 幅が独立計算されて列位置がずれないようにする。
- 操作、badge、可変長テキストの有無で、主要列の開始位置が行ごとに変わらないようにする。
- 列を追加するときは、ヘッダー、loading、データ行の列定義、gap、操作列の扱いが一致していることを確認する。
- tableでは、値を表示する列には対応するヘッダーセルを用意する。アクションボタンやアクションメニューだけの列には見出し文言を置かず、列構造を保つための空のヘッダーセルを用意する。
- 長いテキストは主情報だけ折り返しを許可する。補助情報、状態、操作は折り返しで意味が崩れる場合は縮小・省略・縦積みへ切り替える。
- loadingはskeletonで行数と幅感を保つ。
- emptyは1文の弱いテキストで表示する。empty時にprimary actionを追加する場合は、一覧の外側またはヘッダー右側に置く。
- errorは赤いテキストまたはalert相当で表示し、通常行と同じ見た目にしてはいけない。

初期表示と展開:

- 一覧が補助的なサマリーで、全件表示により主要画面の縦幅が過度に伸びる場合だけ、初期表示件数を制限してよい。
- 展開/折りたたみ操作は低優先度のテキスト寄りボタンにする。
- 展開/折りたたみ操作をbadge、status chip、link風テキストだけで表してはいけない。
- 展開後は同じ操作位置で折りたたみできるようにする。

## 余白

余白は次のscaleを使います。

| scale    | 用途                                                                |
| -------- | ------------------------------------------------------------------- |
| `1`      | ラベルとメッセージ、密接なfield内部。                               |
| `2`      | アイコンとテキスト、詳細ラベルと値、行内の小要素。                  |
| `3`      | フィールド間、ボタン間、一覧行内部の主要gap、ページ内の標準縦積み。 |
| `4`      | セクション間、フォーム本文と操作群、overlay本文の大きな区切り。     |
| `5` 以上 | ページsection同士の大きな分離。通常画面では多用しない。             |

必須ルール:

- ページ直下の主要縦積みは `gap="3"` または `gap="4"` にする。
- フォームのフィールド間は `gap="3"` にする。
- フィールド内部は `gap="1"` にする。
- 詳細表示のラベルと値は `gap="2"` にする。
- ボタン群は `gap="3"` にする。
- 区切り線を使う場合、線の前後に十分な余白を置き、線だけで詰め込まない。
- 画面端の余白はlayoutまたはcontainerに任せ、個別sectionで左右paddingを重ねない。

禁止事項:

- 同じ関係の余白を画面ごとに変えてはいけない。
- CSSで `margin: 13px` のようなscale外の値を作ってはいけない。
- 高さを揃えるためだけに固定heightを置いてはいけない。
- nested cardで余白を稼いではいけない。

例外条件:

- overlayの最大幅、全画面sheet、safe area、viewport heightなど、端末仕様への対応は固定値またはCSS計算を許可する。
- アイコンの視覚補正、ghost系操作の光学調整は、既存UIと同じ目的に限り小さな負marginや局所paddingを許可する。

## ボタン variants

variantは操作の意味を固定するために使います。見た目の好みで選んではいけません。

| 意味         | variant / color               | 用途                                                                                 |
| ------------ | ----------------------------- | ------------------------------------------------------------------------------------ |
| 主操作       | `solid` / 既定accent          | submit、save、create、confirmなど、その文脈で最も進める操作。                        |
| 二次操作     | `soft` / 既定accentまたはgray | 画面遷移を伴う作成起点、filter解除、主操作ではないが見つけやすい操作。               |
| 外部連携操作 | `outline` / 既定色            | 外部プロバイダーでの認証、外部サービス接続など、アプリ内の保存操作ではない開始操作。 |
| 取消         | `soft` / `gray`               | cancel、close、戻すなど、処理を進めない操作。                                        |
| 低優先度操作 | `ghost` / `gray`              | 展開、折りたたみ、編集開始、ナビゲーション、行全体クリック。                         |
| 危険操作     | `solid`または`soft` / `red`   | delete、remove、sign outなど。確定時は強く、導線時はsoftにする。                     |
| icon-only    | icon button / `ghost`         | メニュー、閉じる、テーマ切替など。必ずaria-labelを付ける。                           |

配置:

- フォーム下部の操作群は右寄せを基本にする。
- 取消は主操作の左に置く。
- 破壊的な確定操作は、確認文の後、操作群の右端に置く。
- 行内の編集や削除は行末にまとめる。
- 低優先度の展開操作は中央寄せまたは一覧末尾に置く。

状態:

- loading中の主操作はspinnerを表示し、disabledにする。
- loading中に取消できない仕様なら取消もdisabledにする。
- disabledは見た目だけでなく実際に押せない状態にする。
- icon-only操作は、視覚アイコンだけに依存せず、aria-labelで操作名を明示する。

禁止事項:

- badgeやテキストだけをクリック可能にしてボタンの代わりにしてはいけない。
- 低優先度操作に主操作と同じvariantを使ってはいけない。
- 危険操作を通常accent色で表示してはいけない。
- 1つの文脈に複数の主操作を置いてはいけない。

## フォーム

フォームは次の構造に固定します。

1. 任意のフォーム全体エラー。
2. 入力フィールド群。
3. 任意の補助オプション。
4. 操作群。

フィールド:

- ラベル、入力、メッセージを縦に並べる。
- ラベルはbold、メッセージは小さくする。
- 必須はラベル末尾に赤い `*` を置く。
- 入力の幅は、値の性質に合わせる。短い固定長値や選択値は必要幅、長文や自由入力は横幅いっぱいを基本にする。
- field errorは入力直下に置く。
- 複数メッセージは同じ領域内で改行して表示し、複数の別要素に散らさない。

操作群:

- 操作群は本文から `space-4` 相当で離す。
- 取消と確定を並べる場合、取消を左、確定を右にする。
- 補助オプションがある場合は、補助オプションを左、操作群を右に分ける。
- 送信中は確定操作をloadingにし、多重送信を防ぐ。

エラー:

- フィールドエラーはフィールド単位で表示する。
- サーバーエラーや保存失敗はフォーム全体のalertまたは一時通知で表示する。
- エラー発生時に入力値を消してはいけない。
- 保存成功で閉じるか残すかは、画面の目的に合わせて明示する。連続作成のように残す場合は、継続状態が操作群付近で分かるようにする。

禁止事項:

- ラベルなし入力を作ってはいけない。
- placeholderをラベル代わりにしてはいけない。
- フォーム本文より上に確定ボタンを置いてはいけない。
- 送信中に別フィールドを編集できる状態にしてはいけない。

## Dialog と Overlay

overlayは目的で使い分けます。

| 目的                                     | 表現               | ルール                                                                        |
| ---------------------------------------- | ------------------ | ----------------------------------------------------------------------------- |
| 短い値選択                               | anchored popover   | 入力欄やtriggerに接続し、短い選択だけを扱う。長いフォームや確認文を入れない。 |
| 短い作成/更新/確認                       | centered dialog    | PC/モバイルとも内容が短い場合に使う。                                         |
| 長い入力、詳細編集、複数フィールドの確認 | responsive overlay | PCは最大幅付きdialog、モバイルは全画面sheetにする。                           |
| 破壊的確認                               | confirm dialog     | タイトルを疑問形にし、対象情報と結果を本文に明記する。                        |

共通ルール:

- titleは必須。
- descriptionは、操作結果、対象、注意点、または現在状態を説明する必要がある場合だけ置く。
- close操作はdismissibleなoverlayに必ず用意する。
- モバイルsheetではclose操作を上部右側に置く。
- overlay内の本文は `gap="4"` を基本にする。
- overlay内で区切りが必要な危険操作は、本文の後に区切り線を置いて分離する。
- 短い値選択の一時面は、入力欄にfocusを戻せるようにし、外側clickまたはEscapeで閉じられるようにする。
- 外側クリックやEscapeを止めるのは、編集中の未保存値が失われる、または処理中で状態が壊れる場合だけ。
- 編集中に他の編集操作を開始できないようにする。

サイズ:

- PCの長いoverlayは最大幅 `32rem` を基本にする。
- モバイルsheetは `100dvh`、横幅いっぱい、safe-area込みpaddingにする。
- overlay内で本文が長くなる場合は、画面全体ではなくoverlay contentをスクロール可能にする。

禁止事項:

- 長いフォームを短いconfirm dialogに詰め込んではいけない。
- 短い値選択の一時面に保存ボタン、削除ボタン、複数fieldの編集を入れてはいけない。
- 破壊的操作を通常本文内の主操作と同列に置いてはいけない。
- titleなし、aria-labelなしのoverlayを作ってはいけない。

## レスポンシブ

breakpointは次の意味で扱います。

| breakpoint | 意味                 | 既定                                           |
| ---------- | -------------------- | ---------------------------------------------- |
| `initial`  | モバイルを含む最小幅 | 1列、縦積み、行内完結。                        |
| `sm`       | タブレット以上       | 必要に応じて列分割、ヘッダー表示、操作列分離。 |
| `md` 以上  | 広い画面             | ページ幅の上限維持。過度な横伸びは禁止。       |

必須ルール:

- すべての画面は `initial` で読める順序にする。
- PC向けの列構造をそのままモバイルに押し込んではいけない。
- mobileでは一覧ヘッダーに依存せず、行だけで意味が分かる構成にする。
- 横並びが必要な場合は、折り返し、最小幅、縮小禁止を明示する。
- 長い値がある列は `minWidth=0` 相当を設定し、操作列は縮まないようにする。
- overlayは、長いものをモバイルで全画面sheetにする。
- `display` のresponsive切替でPC専用/モバイル専用の操作を出し分ける場合、同じ操作名と同じ結果にする。

禁止事項:

- モバイルで横スクロールを前提にしてはいけない。ただし表形式で列の意味を保つために明示的に横スクロール領域を設計した場合は例外とする。
- PCで余った横幅を埋めるためだけにカードを増やしてはいけない。
- viewport幅に応じて要素の意味や操作順を変えてはいけない。

## 状態表現

状態は次の表現に固定します。

| 状態             | 表現                                                           |
| ---------------- | -------------------------------------------------------------- |
| loading          | skeleton。ボタン内だけspinner。                                |
| empty            | 弱いテキスト1文。必要なら近くに二次操作。                      |
| field error      | 入力直下の小さい赤テキスト。                                   |
| page/list error  | 赤テキストまたはalert相当。                                    |
| save error       | フォーム全体alertまたは一時通知。                              |
| success          | 一時通知。画面内に恒久的な成功文を残さない。                   |
| disabled         | 見た目と実操作の両方をdisabledにする。                         |
| selected/current | 状態表示として明確にする。操作可能に見せる場合はbuttonにする。 |

禁止事項:

- loading中にレイアウトを大きく変えてはいけない。
- emptyをerror色で表示してはいけない。
- errorをempty文言で代替してはいけない。
- 成功通知をエラーと同じ位置・同じ色で出してはいけない。

## 既存例外の扱い

既存コードにこの文書と完全に一致しないUIがある場合、次のように扱います。

- 既存例外は、新規UIの前例にしてはいけない。
- 例外を触る変更では、この文書に寄せる。ただし、要求範囲を超える大きな改修になる場合は、Design / Planで別タスク化する。
- 固定height、直接style、独自CSS値、古いコメント、未使用tokenを見つけた場合、それを根拠に同じ実装を増やしてはいけない。
- 既存例外を維持する場合は、Requirements / DesignまたはPR説明に「なぜ今回直さないか」を記載する。
- 古い汎用surfaceで固定pxのpadding、独自shadow、token外の角丸が使われている場合、新規画面では採用しない。触る場合はテーマtokenと既存surface primitiveへ寄せる。
- 古い折りたたみUIで固定height、独自accent色、手書きanimationが使われている場合、新規の一覧開閉には採用しない。触る場合は低優先度ボタンによる展開/折りたたみ、またはUIライブラリの公式primitiveへ寄せる。

## 実装前チェック

AI agent はUI実装前に次を確認します。

- 画面は入力、閲覧、一覧、詳細、確認、設定のどれか。
- 一覧はカード、管理行、table、key-value、状態集合のどれか。
- 文字階層はページタイトル、セクション見出し、ラベル、標準本文、詳細値、強調値、補助情報、エラーのどれか。
- 操作は主操作、二次操作、取消、低優先度、危険操作、icon-onlyのどれか。
- 余白は `1` から `4` のscaleで説明できるか。
- mobile `initial` で読み順が成立するか。
- loading、empty、error、disabled、submitting、successの状態が定義されているか。
- ドメイン固有の値順、表示形式、文言は別ルールに従っているか。
