# Domain UI Rules

この文書は、ドメイン固有のUI判断で守るべき意味のルールです。

UIの土台、文字階層、余白、variant、フォーム、overlay、responsiveは [Web Design Rules](./design-rules.md) に従います。この文書は、画面にどの情報を持たせるか、どの項目をどの順番で表示するか、どのコンポーネントで実装するかを固定しません。

AI agent は、ドメイン値を扱うUIを作るとき、この文書を「混同してはいけない意味」と「Design / Planで決めるべき判断対象」の基準として使います。具体的な表示順、入力順、列構成、件数、文言は、Requirements / Designで決めます。

## 共通原則

- ドメイン値は、見た目の部品ではなく、ユーザーの判断や操作結果に影響する意味として扱う。
- ドメイン値をUIに出す前に、その値でユーザーが何を判断したいのかをDesignで明確にする。
- ドメイン値の種類名だけで表示内容を決めてはいけない。予算、前年値、前月値、残高、見込み額、日付、カテゴリ、状態などの名前は、ユーザーの判断目的を説明しない。
- 値の有無、値が0であること、参照先が不明であること、削除済みであること、未設定であることを混同してはいけない。
- ユーザーが入力した値と、システムが付与した状態や説明文を混ぜて1つの文言にしてはいけない。
- 表示形式は、同じ種類の値では画面ごとに揺らさない。
- ドメイン制約に反する状態は、汎用エラーではなく、何が制約に反しているか分かる形で示す。
- 1回の操作として見える保存や更新は、完了後も1つの結果として理解できるようにする。
- 部分的な成功、古い値、失敗後の一時状態が、完了済みの状態に見えてはいけない。

## ドメイン値の利用目的

ドメイン値は、種類ではなく利用目的でUI上の扱いを決めます。

Design / Planでは、ドメイン値ごとに次のどれを目的にするかを明示します。

- 実値を知りたい
- 比較したい
- 残量や到達可否を知りたい
- 制約に違反していないか知りたい
- 内訳を確認したい
- 対象を識別したい

この文書では、各値の分類結果や具体UIを固定しません。分類結果、値そのものを表示するか、判断結果だけを表示するか、比較元や基準値を添えるかは、Requirements / PRDとDesign Docで決めます。

必須ルール:

- 値の種類名だけでUIを決めてはいけない。
- 値ごとの利用目的がRequirements / PRDまたはDesign Docにない場合、UI詳細を決めてはいけない。
- 目的が比較なら、比較元そのものを主役にするかどうかをDesign Docで判断する。
- 目的が実値確認なら、判断結果ではなく値そのものを主役にするかをDesign Docで判断する。
- 目的が残量や到達可否の確認なら、値そのものより結果や状態を主役にするかをDesign Docで判断する。
- 目的が複数ある場合は、主目的と補助目的を分けてDesign Docに残す。

## 金額

金額は、入力、表示、集計、基準、比較結果として扱われやすい値です。

- 金額は、数値入力、表示値、集計値、基準値、比較結果のどれとして扱うのかをDesignで明確にする。
- 金額が「未設定」なのか「0」なのかを必ず区別する。
- 金額を主要な判断材料にする画面では、他の補助情報より強く見えるようにする。
- 金額を補助的に見せる画面では、主要情報より強くしてはいけない。
- 集計値は、元の明細値と区別できる文脈に置く。
- 通貨表記、桁区切り、負値や超過状態の扱いは画面ごとに変えない。
- 差額は、基準値と実績値の比較結果として扱い、単なる符号付き金額として表示してはいけない。
- 差額を表示する場合は、何に対する差なのか、残額なのか超過額なのかをUI上で区別できるようにする。
- 差額が0の場合は、残額、超過額、一致状態のどれに含めるかをDesignまたは該当ドメインルールで決める。
- 基準値が未設定、計算不能、取得失敗の場合は、差額を0として表示してはいけない。

Design / Planで決めること:

- 金額が入力値、実績値、集計値、基準値、比較結果のどれか。
- 金額が主情報か補助情報か。
- 金額の比較対象があるか。
- 比較対象を表示する必要があるか、比較結果だけで足りるか。
- 差額0を残額、超過額、一致状態のどれに含めるか。
- 0、未設定、計算不能、取得失敗をどう区別するか。
- 入力時に許可する値の範囲、単位、丸め、エラー文言。

## 日付と月

日付と月は、対象期間、履歴、制約判定に使われる値です。

- 日付と月を同じ粒度の値として扱ってはいけない。
- 表示対象の期間と、入力または変更できる期間を混同してはいけない。
- 過去、現在、未来のどれが許可されるかは、UIだけで推測させず、制約として明確にする。
- 現在日付や現在月に依存するUIは、実行時の現在値に対して一貫した表示とvalidationを行う。
- 日付や月がフィルター条件なのか、レコード自身の属性なのか、履歴の有効開始なのかをDesignで明確にする。

Design / Planで決めること:

- 日付と月のどちらを扱うか。
- 値が対象期間、作成日、更新日、有効開始、検索条件のどれか。
- 過去、現在、未来の許可範囲。
- 値が未選択、範囲外、不正、取得不能の場合の表示。

## 分類と状態

分類や状態は、ユーザーが情報を探す、集計を読む、対象を識別するための補助概念です。

- 分類名と状態ラベルを混同してはいけない。
- 状態ラベルは操作ではない。操作できる場合は、状態表示ではなく操作要素として表す。
- 優先表示、固定表示、非表示、削除済み、不明などの状態は、通常値と同じ意味に見えてはいけない。
- ユーザーが変更できる分類・状態と、システムが決める状態は区別する。
- 参照先が削除済みまたは取得不能でも、元のレコード自体が消えたように見せてはいけない。

Design / Planで決めること:

- 分類が主情報か補助情報か。
- 状態が読み取り専用か、ユーザーが変更できるものか。
- 優先表示や固定表示がある場合、その理由と解除方法。
- 削除済み、不明、未設定、取得失敗の表示差分。

## 明細、集計、基準値

明細、集計、基準値は、同じ数値や分類を扱っていても意味が異なります。

- 明細は個別の事実として扱う。
- 集計は複数の明細から計算された結果として扱う。
- 基準値は判断のための設定値として扱う。
- 明細、集計、基準値を同じ表示階層に置く場合は、何が事実で何が計算結果で何が設定値か分かる文脈を作る。
- 集計が空であること、計算できないこと、取得に失敗したことを混同してはいけない。
- 基準値が未設定であることと、基準値が0であることを混同してはいけない。

Design / Planで決めること:

- 画面の主目的が明細確認、集計確認、基準値管理のどれか。
- 集計対象がない場合に表示を省略するか、empty状態を見せるか。
- 基準値の未設定、解除、削除、0設定をどう区別するか。
- 明細更新後に集計や基準値との関係をどう更新して見せるか。

## 作成、更新、削除

ドメイン操作は、ユーザーが理解する1回の結果を基準に設計します。

- 作成は、新しい対象がどの範囲に反映されるか分かるようにする。
- 更新は、変更前後で何が変わったかを誤解させない。
- 削除は、対象自体が消えるのか、参照できなくなるのか、履歴は残るのかを区別する。
- 削除や解除は、未設定化、非表示化、0設定と同じ意味にしてはいけない。
- 保存失敗後に、UIだけが成功状態に進んだように見えてはいけない。
- 連続作成やinline編集など、画面を閉じない操作では、次に入力・編集している対象が分かるようにする。

Design / Planで決めること:

- 操作成功後に閉じるか、残るか、リセットするか。
- 失敗時に入力値や編集中の値を維持するか。
- 削除後に関連する表示をどう扱うか。
- 1回の保存に複数のドメイン変更が含まれる場合、部分成功をどう防ぐか、またはどう表示するか。

## 画面種別ごとの判断

画面種別ごとに、以下の判断をDesign / Planで明示します。

### 一覧

- 一覧の主目的が、検索、比較、履歴確認、選択、管理のどれか。
- 1行で主情報にする値と、補助情報にする値。
- 行全体が操作なのか、行内の個別操作が主なのか。
- empty、loading、error、不明参照の扱い。

### 詳細

- 詳細が読み取り専用か、編集可能か。
- 編集できる値と、参照のみの値。
- 複数の値を同時編集できるか、1つずつ編集するか。
- 編集中に閉じる、切り替える、別操作をする場合の未保存値の扱い。

### フォーム

- 入力対象の値が必須か任意か。
- 値が未入力、初期値、既存値、リセット値のどれか。
- 保存時に複数のドメイン値を同時に変更するか。
- 送信中、成功、失敗、連続操作の状態。

### サマリー

- サマリーが現在状態、期間状態、集計結果、基準値との差分のどれを示すか。
- 詳細を見る前に必要な最小情報。
- 内訳を見せる必要があるか、初期状態では省くべきか。
- 集計対象なし、基準値なし、取得失敗の違い。

### 設定

- 設定が現在値の変更なのか、分類や基準値の管理なのか。
- 設定変更が既存データに与える影響。
- 削除や解除が既存表示に与える影響。
- 設定値が他画面でどのように反映されるか。

## 禁止事項

- ドメイン固有の表示順や入力順を、この文書だけを根拠に固定してはいけない。
- 「既存画面がそうだから」という理由だけで、異なる目的の画面に同じ情報構造を流用してはいけない。
- 画面に必要な値を、実装しやすいデータ構造の順番で並べてはいけない。
- ユーザーが判断するための値を、loading、empty、error、不明、未設定の状態と同じ見た目にしてはいけない。
- ドメイン制約をDBエラーだけに任せ、UIで選択不能または理解可能にする判断を省いてはいけない。
- 破壊的な操作を、通常の更新や解除と同じ意味に見せてはいけない。

## 実装前チェック

- 画面の目的は、明細確認、集計確認、設定管理、作成、更新、削除のどれか。
- 扱う値は、事実、集計結果、基準値、分類、状態、検索条件のどれか。
- 値なし、0、不明、削除済み、未設定、取得失敗を区別しているか。
- 表示順、入力順、列構成をRequirements / Designで決めているか。
- その順番は、データ構造ではなくユーザーの判断順に基づいているか。
- 保存や削除の結果が、ユーザーにとって1回の操作として一貫して見えるか。
- 関連する他画面の表示や集計が古い状態に見えないか。
